## 1. 배경

`/opic-study` 세션 룸에서 멤버가 답변하면, `opic-study-feedback` EF가 Whisper STT + Azure 발음 평가 + GPT-4.1-mini 코칭을 종합해 **8개 섹션 코칭 노트**를 생성한다(`feedback_result` JSONB). 멤버 전원이 함께 보며 토론하는 자료다.

이번 점검에서 5/26~6/1 실제 스터디 답변 **122건(정상)** 을 전수 분석했다. 파이프라인 신뢰성은 양호(에러 0건)하나, **콘텐츠 품질·구조에 1개 치명적 버그 + 3개 품질 이슈**를 발견했다.

## 2. 진단 (데이터 근거)

### 🚨 P0 — `flow` 섹션 소실 (타입 불일치 버그)

EF의 검증 코드가 `flow`를 **문자열**로 취급해 GPT가 반환한 **객체**를 빈 문자열로 덮어쓴다.

```ts
// supabase/functions/opic-study-feedback/index.ts:495 — 🐛
parsed.flow = typeof parsed.flow === "string" && parsed.flow ? parsed.flow : "";
```

반면 나머지 전 계층은 객체로 정의:
- **프롬프트**: `"flow": { "intro": "...", "body": "...", "conclusion": null }` 출력 지시
- **타입** (`lib/types/opic-study.ts:172`): `flow: { intro, body, conclusion }`
- **UI** (`SessionReplay.tsx:441`): `feedback.flow.intro / .body / .conclusion` 읽음

→ GPT 객체 반환 → `typeof === "string"` 거짓 → `""` 덮어씀 → UI 조건 falsy → **흐름 섹션 영구 미표시**.

**데이터 증거**: `flow = ""` 113건 vs 객체 보존 9건(구버전 배포분).
**영향**: 5/26 이후 전 멤버가 "답변 흐름" 코칭을 한 번도 못 봄.

---

### ⚠️ P1-a — 등급 추정 IM3 쏠림 (앵커링)

| 등급 | 건수 | 비율 |
|------|------|------|
| **IM3** | **82** | **67%** |
| null | 23 | 19% |
| IH | 11 | 9% |
| IM1 / IM2 / IL | 3 / 2 / 1 | 5% |

transcript 길이가 530~1,300자로 다양한데도 67%가 IM3로 수렴. 프롬프트의 *"너무 후하지도 인색하지도 않게"* 가 GPT를 안전한 중앙값으로 끌어당기는 전형적 앵커링. 매번 같은 등급 → 성장 피드백으로서 변별력 약함.

### ⚠️ P1-b — 답변자 목표 등급 미활용

EF가 `answerer_target_grade`를 프롬프트에 주입(`index.ts:425`)하지만 **system 프롬프트가 이 값을 피드백 방향 조절에 전혀 안 씀**. IH 목표 학습자와 IM2 목표 학습자가 동일 톤·깊이의 코칭을 받음. 개인화 핵심 입력이 사실상 버려짐.

### ⚠️ P2-a — 발음 패턴 제네릭 양산

샘플 다수가 *"프로소디가 평탄해서 문장 끝 억양…"* 으로 동일 수렴. Azure word-level 약점 단어(상위 5개)가 프롬프트에 전달되나, 출력은 단어 특이성 없이 일반론. 112/122가 패턴을 생성 — *"패턴 없으면 빈 배열"* 지시 미준수.

### ⚠️ P2-b — `refine_expressions` 우선순위 없음

3개를 나열하나 OPIc 등급 영향이 큰 순서(시제 일관성·문장 연결 등)가 아님. 학습자가 "무엇부터 고칠지" 알기 어려움. (단, 오류 탐지 정확도 자체는 우수 — 유지.)

### ℹ️ P3 — `max_tokens` 여유 부족

8섹션 풀세트 + 한글 출력 시 `max_tokens: 2000`(`index.ts:460`)에 근접. `finish_reason: length` → JSON 파싱 실패 위험. 현재 에러 0건이나 예방적 상향 권장.

## 3. 개선안 상세 (before → after)

### P0. flow 버그 수정 ⭐ 최우선

**파일**: `supabase/functions/opic-study-feedback/index.ts:495`

```ts
// AS-IS (🐛)
parsed.flow = typeof parsed.flow === "string" && parsed.flow ? parsed.flow : "";

// TO-BE
parsed.flow =
  parsed.flow && typeof parsed.flow === "object"
    ? {
        intro: typeof parsed.flow.intro === "string" ? parsed.flow.intro : null,
        body: typeof parsed.flow.body === "string" ? parsed.flow.body : null,
        conclusion: typeof parsed.flow.conclusion === "string" ? parsed.flow.conclusion : null,
      }
    : { intro: null, body: null, conclusion: null };
```

- `FeedbackOutput.flow` 타입(`index.ts:46`)은 이미 객체 → 타입 정합.
- UI/타입 변경 불필요. EF만 수정 후 재배포.
- 기존 113건은 재생성 대상 아님(과거 세션). 수정 시점 이후 신규 노트부터 흐름 표시.

### P1-a. 등급 추정 앵커링 완화

**파일**: `evaluation_prompts.opic_study_feedback` (system) — `## 8. estimated_level` 섹션

추가/수정할 지시(요지):
- *"답변을 먼저 IM3로 가정하지 말 것. transcript의 **문장 구조 복잡도·어휘 다양성·내용 전개**를 독립적으로 채점한 뒤 등급을 정하라."*
- 등급별 정량 신호를 더 또렷하게:
  - 복문/부사절 **3회 이상 + 추상 표현** → IH 이상 적극 고려
  - 단문 위주 + 복문 1~2회 → IM2~IM3
  - 분절적·반복 빈번 → IM1 이하
- *"확신 없으면 IM3로 도피하지 말고, 가장 가까운 두 등급 중 근거가 우세한 쪽을 택하라."*

> 목표: IM3 비율을 67% → 합리적 분산(예: IM2~IH 골고루)으로.

### P1-b. 목표 등급 캘리브레이션

**파일**: system 프롬프트 + (선택) user 프롬프트 활용 강화

- system 프롬프트에 *"답변자의 목표 등급({answerer_target_grade})을 기준으로 `next_speaker_tip.enhance`와 `estimated_level.next_level_tip`을 작성하라. 현재 답변과 목표 등급 사이의 gap을 한 가지 핵심 행동으로 제시."* 추가.
- 예: 목표 IH인데 현재 IM2 → *"IH로 가려면 부사절(although/while) 2개 + 이유 설명 1문장 추가"*.
- **선결 과제**: 멤버 `profiles.target_grade` 실제 설정 여부 확인(현재 대부분 미설정 → 기본 AL). 미설정 시 캘리브레이션 효과 제한적이므로, 멤버 온보딩에서 목표 등급 입력 유도 병행 검토.

### P2-a. 발음 패턴 구체화 + 빈 배열 강제

**파일**: system `## 5. pronunciation_patterns` + user 프롬프트

- *"Azure 약점 단어가 전달되면 그 단어를 인용해 구체화하라(예: 'thirty', 'through'의 th가 약함). 일반론('프로소디 평탄') 단독 출력 금지."*
- *"뚜렷한 패턴이 없으면 빈 배열 `[]`. 억지로 만들지 말 것."*
- user 프롬프트의 `{pronunciation_text}`에 약점 단어가 이미 포함됨 → 활용 강조.

### P2-b. refine_expressions 우선순위화

**파일**: system `## 4. refine_expressions`

- *"OPIc 등급에 영향이 큰 순서로 정렬: ① 시제 일관성 ② 문장 연결/복문 ③ 한국식 표현 ④ 관사·전치사 등 미세 오류. 가장 임팩트 큰 것을 첫 번째에."*
- 개수 가이드 유지(1~3개).

## 부록 — 강점(유지할 것)

- `refine_expressions`는 `these kind of`, `get all of things`, 과거시제 오류 등 **학습자가 한 번 듣고 못 잡는 미묘한 오류를 정확히 포착**. 코칭 노트의 핵심 가치 — 구조·정확도 그대로 유지.
- 파이프라인 안정성(에러 0/127), 무료 정책, 점수 비공개 정책 모두 정상 동작.
