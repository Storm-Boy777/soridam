## 0. 한눈에 보기 (TL;DR)

> **제품 한 줄:** 소리담 = **AI 일타강사.** 실제 시험 리포트 + 목표 등급을 받아 *"뭘·어떻게 학습해야 개선되는지"*를 1:1로 처방하고 훈련시켜 **원하는 등급까지** 끌어올린다.

- **문제** — SAIL/OPIc 리포트는 *진단서일 뿐 처방전이 아니다.* 학습자는 "약점은 알겠는데 **어떻게** 고치지?"에서 막힌다(사용자 본인 실증). 등급은 올리고 싶은데 방법을 모른다.
- **해결** — **진단 → 행동 다리**: 약점을 "오늘 실행 가능한 학습 행동"으로 번역. *방법*은 큐레이션(일타강사 방법론 라이브러리), *콘텐츠*는 본인 답변 기반 개인화 **생성**(RAG). → §4.1
- **이미 있는 것** — 모의고사 평가 · 세부진단서 42코드 · 튜터링(진단→처방→훈련 루프) · 코치/쉐도잉 **모두 구현됨**. SAIL 리포트도 이미 소비 중(단, 1인 정적 하드코딩). → §3
- **신규 = 그 위의 3가지:**
  1. **통합 `ProficiencyReport` 스키마** — 실제 SAIL 리포트 ≡ 소리담 모의고사 리포트 (§4.2)
  2. **모의고사 출력을 SAIL식 리포트로 업그레이드** — 소리담 모의고사는 이미 *SAIL-shaped*. 이것이 **데이터/볼륨 엔진** (§4.4·§4.5)
  3. **목표 등급 트레이너** — 기존 자산을 목표에 묶어 *생성·오케스트레이션* (중복 OK, 고도 다름) (§4.1)
- **왜곡 방어** — "리포트 1장 = 진실" 함정은 **데이터 축적**(native 모의고사 볼륨 + 실제 리포트·달성등급 앵커)으로 (§4.3)
- **MVP** — 통합 스키마 + 모의고사 SAIL식 리포트(마이크로 문법 진단 + 본인 답변 인라인 교정) (§6)
- **미결 5건** — UI 위치 / 스키마 저장 / 외부 임포트 범위 / 등급 천장 / 멤버 데이터 (§7)

## 1. 배경 — SAIL과 Proficiency Report

**SAIL (Samsung AI Language Test)** = 멀티캠퍼스 운영, ACTFL 출제, OPIc을 **AI 머신러닝 자동 채점**으로 발전시킨 비공인(사내용) 영어 말하기 시험. 20년치 사람 채점 데이터 학습.

### SAIL vs OPIc 핵심 차이
| 구분 | SAIL | OPIc |
|------|------|------|
| 채점 | ACTFL AI Machine Scoring (AI 2회 + 불일치 시 3차) | Certified Rater(사람) |
| 공인 | 비공인(사내용) | 공인 |
| 등급 범위 | **NL~S** (AM·AH·S 추가) | NL~AL |
| 인터랙션 | 답변 후 **Ava's Reaction**(추임새) → 인터뷰 경험 | 단순 청취·녹음 |
| 결과 회신 | **24시간** | 7일 |
| **피드백** | 등급 + **Proficiency Report** | 등급만 |
| 등급 세분화 | Intermediate Mid → **Mid 1/2/3** | IM1/2/3 |

> **OPIc 대비 SAIL의 본질적 차별점은 단 하나 — "Proficiency Report"라는 진단 피드백을 준다는 것.** (사용자 코멘트)

### Proficiency Report 구조 (3계층)
실측 리포트(전진성)에서 추출한 스키마 — **그대로 데이터 모델이 됨**:

| 계층 | 항목 | 비고 |
|------|------|------|
| **① 요약** | SUMMARY STRENGTH / SUMMARY WEAKNESS (각 1문장) | 헤드라인 |
| **① 상세** | STRENGTH / WEAKNESS (불릿 4~5개) | |
| **② FACT 4기준** | FUNCTIONS AND TASKS · ACCURACY · CONTEXT AND CONTENT · TEXT TYPE | 각각 `can do` + `not yet able to` 쌍 |
| **③ 마이크로 문법** | TENSE · SUBJECT-VERB AGREEMENT · ARTICLES · PLURAL | 구체적 오류 패턴 |

핵심: **②의 `can do / not yet able to` 프레이밍**은 ACTFL sublevel 논리 그대로 — "지금 유지하는 능력 vs 다음 등급을 규정하는 능력". 즉 리포트가 **"현재 등급 → 다음 등급"으로 가는 체크리스트를 이미 생성**해준다.

## 2. 핵심 통찰

> **SAIL은 "진단서"에서 멈춘다. 소리담은 "학습 플랫폼"이다.**
> SAIL은 "당신은 X를 아직 못 한다"를 정확히 짚지만 **"어떻게 고치는지"와 "연습 거리"는 0**이다. 그 빈칸이 정확히 **진단 → 처방 → 훈련 → 재측정** 루프 — SAIL이 못 닫는 고리를 소리담이 닫는다.

> **결정적 페인 (사용자 실증):** 리포트를 받아도 학습자는 *"그래서 이 약점을 어떻게 고치지?"*를 모른다. 등급은 올리고 싶은데 **방법(method)을 모른다.** → 신규 기능의 본질 = **진단을 "오늘 실행 가능한 개선 방법"으로 번역**하는 것. (§4.1 ▸ 개선 방법론 라이브러리)

**보강 (2026-06-06 사용자 방향):** §3에서 보듯 소리담은 진단·처방·훈련 자산을 이미 갖췄다. 신규 기능은 이것들을 **버리거나 피하는 게 아니라**, **하나의 목표(원하는 등급)에 묶어 오케스트레이션하는 상위 계층**이다. 기존 자산과의 기능 중복은 문제되지 않는다 — 이들은 **재료**이고, 신규 기능은 **목표에 연결된 레시피**다 (고도(altitude)가 다름, §4.1). 단, "리포트 1장 = 진실"이라는 함정만은 **데이터 전략**으로 방어한다(§4.3).

## 4. 신규 기능 — 리포트 기반 "목표 등급 트레이너" (확정 방향)

> **북극성 (사용자):** 실제 시험 리포트를 바탕으로, 학생이 **원하는 등급을 리포트 기반으로 훈련하여 받게** 한다.
> 기존 소리담 자산을 적극 동원하고, **기능 중복은 허용**한다. 병행 목표: **모의고사 출력을 SAIL 리포트 형태로 업그레이드.**
>
> **🎯 한 줄 정의 = "AI 일타강사".** 학생이 실제 리포트를 들고 와 *"목표가 AL인데 어떻게 해요?"* 물으면, 일타강사가 1:1로 *"당신은 이걸 먼저 / 이건 외우고 / 이 틀은 몸에 익을 때까지"* 처방하듯 — **그 컨설팅을 시스템이 재현**한다. (학습자가 가장 목말라한 것 = "뭘·어떻게 해야 내 문제가 개선되나")

### 4.1 핵심 흐름 (6단계) — 목표 지향 오케스트레이션 + 추적 계층
신규 기능은 기존 자산을 *대체*하지 않고 그 **위에 얹혀 목표에 묶는** 계층이며, 핵심 동작은 **이 사람만을 위한 훈련의 *생성*(라우팅 아님)**이다.

1. **리포트 입력** — ① 실제 SAIL/OPIc 리포트(붙여넣기·임포트) 또는 ② 소리담 모의고사 native 리포트. **둘 다 동일 스키마(§4.2)로 정규화.**
2. **목표 등급 설정** — 현재 → 원하는 등급 (예: IH → AL, IM2 → IH). `target_grade`는 이미 멤버 속성으로 존재.
3. **갭 분석** — 리포트의 `not yet able to` + 마이크로 문법 패턴 → §5 매핑표로 **현재→목표 사이의 구체적 블로커**로 변환.
4. **처방 = 개인 훈련 *생성*** — 고정 콘텐츠 배정(라우팅)이 **아니라**, 블로커별로 **이 사람 전용 훈련을 생성**. 소리담 재료(질문 DB·답변뱅크·본인 transcript)에 묶인 **그라운디드 생성(RAG)**. 기존 모듈(튜터링·시제 아크·쉐도잉·코치)은 *재료이자 실행 엔진*으로 호출. **중복 OK.** (방법은 §4.1 ▸ 참조)
5. **훈련 → 재측정** — 학생이 훈련 → 모의고사 재응시 → 새 리포트 → 갭 축소. `not yet able to` → `can do` 전환을 추적.
6. **데이터 축적** — 매 회차 리포트가 데이터셋에 적립(§4.3).

> **고도(altitude) — 튜터링과의 관계:** 튜터링은 *병목 기반·단기 전술 엔진*. 신규 트레이너는 *목표 등급 기반·장기 전략 로드맵*으로, **튜터링을 하나의 실행 단계로 호출**한다. 같은 일이 아니라 **다른 층위**다. (현재 `/coaching/curriculum`의 4주 IH 커리큘럼이 이 트레이너의 **정적 프로토타입** — 신규 기능은 이를 **동적·다(多)등급·리포트 구동**으로 일반화.)

#### 4.1 ▸ 진단 → 행동 다리: '개선 방법론 라이브러리' ⭐ (제품의 심장)

> **사용자 실증 페인:** 리포트를 받아도 *"그래서 이 약점을 어떻게 개선하지?"*를 모른다. 등급은 올리고 싶은데 **방법(method)을 모른다.** 의욕 있는 상급 학습자조차 막힘 = **보편 페인.** 리포트는 진단서일 뿐 처방전이 아니다.

**빠진 고리 3개:** 진단 → ❶메커니즘(왜 약점인가) → ❷방법(어떻게 고치나) → ❸오늘의 행동(뭘 하나) → 확인. 신규 기능이 이 셋을 채운다.

**핵심 페다고지:** 말하기 개선 ≠ 문법 규칙 암기. **인지(noticing) → 통제 연습 → 자동화 → 전이(spontaneous)** 4단계. (튜터링 `Q1학습→Q2적용→Q3독립` 드릴이 이미 이 구조 — *표면화·설명·리포트 구동*만 추가하면 됨.)

**일타강사의 교수 동작 = 기능 명세 (1:1 컨설팅 재현):**
1. **진단 읽기** — 리포트를 "당신에게 이게 진짜 무슨 뜻인지"로 번역
2. **우선순위** — "당신은 *이걸 먼저*" (단 하나, 등급 레버)
3. **처방 분류** — **외울 것**(고정 청크·신호어) / **틀 연습**(스켈레톤, *몸에 익을 때까지*) / **의식할 것**(속도 속 정확도). ← 약점마다 학습 *행동*이 다름 (사용자 직감 그대로)
4. **왜 설명** — "문단이 안 나온다 = *틀이 없다*는 것" (납득시켜 행동 유도)
5. **시범** — 본인 답변의 *목표 등급 버전* 제시
6. **반복 + 점검** — "몸에 익을 때까지" + 다음 시험에서 확인
> ⚠️ **절제 원칙:** 한 번에 하나(코치 `ONE weakness at a time`). "다 고치세요" ❌ — 일타강사는 욕심내지 않는다.

**아키텍처 — 방법(curated) × 콘텐츠(generated):**
- **개선 방법론 라이브러리** = 약점 클래스별 *검증된 고치는 법* (유한·큐레이션 → 신뢰 가능, 할루시네이션 ❌)
- **개인 콘텐츠 생성** = 그 방법 *안을* 본인 오류·답변·토픽으로 채움 (RAG)

| 약점 클래스 | 개선 방법 (검증된 페다고지) | 동원 자산 |
|---|---|---|
| 관사·복수·수일치 | 본인 오류 인지 → 통제 정확도 반복 → 속도 재현 (규칙암기 ❌) | transcript·질문DB |
| 시제 혼용 | 시간축 서사로 프레임 고정 | 시제 만능 아크 |
| 문단 전개·주제 유지 | 스켈레톤 내재화 + 본인 답변 재구성 | 258룰·skeleton·답변뱅크 |
| 유창성·전달 | 모범 발화 쉐도잉 | 쉐도잉 4-Step |

**등급 레버 (우선순위):** "뭘 먼저?"의 답. 코치 개입 우선순위(P1 담화 > … > P6 문법) + 4 IH 기둥 적용 → *"AL 가려면 관사가 아니라 담화 구조가 레버. 이것부터."* "약점 다 고쳐"가 아니라 **"등급 올리는 단 하나부터."**

**확인:** 재측정으로 지표 이동을 보여줌(관사 41%→22%, `not yet`→`can do`) = *"내 방법이 통한다"* 의 증거.

### 4.2 통합 리포트 스키마 (interlingua) — 가장 중요한 설계 결정
실제 SAIL 리포트와 소리담 모의고사 리포트가 **하나의 canonical `ProficiencyReport` 스키마**로 수렴해야 한다(SAIL 3계층 채택). 그래야:
- 트레이너가 **출처와 무관하게** 동일 입력을 읽는다.
- 외부 리포트 임포트 ≡ native 모의고사 리포트 (상호 교환 가능).
- 데이터셋이 **동질적** → 캘리브레이션 가능.

```
ProficiencyReport {
  summary { strength, weakness }
  fact { F, A, C, T } { can_do, not_yet }
  micro_grammar { tense, sva, article, plural } { pattern, severity, freq% }
  meta { source: 'sail'|'opic'|'soridam_mock', level, level_confidence(±1), achieved_grade? }
}
```

### 4.3 데이터 전략 — "리포트 1장 = 왜곡" 방어 (사용자 우려 반영)
사용자 지적 정확: 단일 리포트는 한 시점·한 시험지·한 채점 패스의 스냅샷 → **표본 분산 + 1인 오류 프로파일 과적합** 위험.

| 데이터 소스 | 역할 | 특성 |
|------------|------|------|
| 소리담 모의고사 native 리포트 | **볼륨 엔진** | 즉시·저비용·매 응시 1건. 단일 왜곡을 *다회차 궤적*으로 평탄화 |
| 실제 SAIL/OPIc 리포트 임포트 | **그라운드 트루스 앵커** | 외부 검증. 모의고사 정확도를 실측에 캘리브레이션 |
| 실제 달성 등급(achieved_grade) | **골드 라벨** | "처방대로 훈련 → 진짜 목표 등급 받았나" 검증 |

→ 전략: **native 모의고사로 볼륨 확보 + 실제 리포트/달성등급으로 보정.** 사용자가 제안한 "재응시·타 멤버 결과 수집"이 정확히 **앵커 데이터**. 1인(전진성) 하드코딩 매핑(`WEAKNESS_MAP`)은 데이터가 쌓이면 **모집단 기반 동적 매핑**으로 교체.

### 4.4 "소리담 모의고사는 SAIL에 더 맞다" — 동의 (+ 정밀화)
사용자 통찰 정확. 변별 축:

| | 채점 | 출력 |
|---|------|------|
| OPIc(실제) | 사람 Rater | **등급만** (+요청 시 종이 진단서) |
| SAIL | AI | 등급 + **Proficiency Report** |
| **소리담 모의고사** | **AI(EF)** | 등급 + **coaching_report 9섹션 + FACT** |

→ 소리담 모의고사는 구조적으로 **OPIc이 아니라 SAIL을 복제**한 형태 — *born SAIL-shaped*. 정밀화: **문항/포맷은 OPIc 기반, 채점·리포트 아키텍처는 SAIL식**인 하이브리드. 따라서 업그레이드(=출력을 SAIL 리포트 형태로)는 **피벗이 아니라 출력 계층의 완성**. 포지셔닝: 'SAIL 대비 연습' 각도도 가능하되, **OPIc 응시자에게도 유효한 공통 리포트** 유지가 전제.

### 4.5 리포트 엔진 업그레이드 — 모의고사 출력을 SAIL 리포트로 (8개 갭)
4.1~4.4를 가능케 하는 **엔진**. `coaching_report` 출력을 SAIL 수준 진단서로 끌어올린다:

| # | 갭 | 현황 | 목표(SAIL 수준) | 손볼 곳 |
|---|----|------|----------------|---------|
| 1 | **grammar_type 카테고리화** | `error_parts` free-text("관사 the 누락") | TENSE/ARTICLE/SVA/PLURAL/PREP 태깅 | EF Stage B-1 eval 또는 Stage C 파싱, `mock_test_evaluations.corrections`에 `grammar_type` 추가 |
| 2 | **grammar_patterns 정량화** | 없음 | `{ARTICLE:41%, TENSE:23%, SVA:8%, PLURAL:6% …}` | report 집계 로직 |
| 3 | **FACT 차원별 micro breakdown** | `score_f/a/c/t` 총점만 | `fact_micrograms:{A:{weak_ckboxes, severity, reason}}` | coaching_report 확장 |
| 4 | **등급 한 줄 해석** | 없음 | `"IM2: 문장을 안정적으로 이어갑니다"` | `proficiency_interpretation` (등급별 템플릿, 전문가피드백 문서에 권장안 존재) |
| 5 | **병목 분석** | top3_priorities는 있으나 축 귀속 약함 | `bottleneck_analysis:{primary_axis:'A', reason, next_level_potential}` | rule-engine + report |
| 6 | **신뢰구간** | `final_level` 단일값 | `IM2(±1)` — Floor/Ceiling 분포 기반 | `reports.final_level_confidence` |
| 7 | **다음 등급 예시 답변** | roadmap 추상적("문단 구조 필요") | `next_level_example_answer`(IM2 vs IM3 실제 발화 대조) | 쉐도잉 답변뱅크 재활용 |
| 8 | **성장 마이크로 추적** | growth_analysis는 FACT 수치 비교만 | `grammar_change:{TENSE:+3%, ARTICLE:-2%}` 회차 델타 | growth_analysis 확장 |

> **차별화(SAIL 능가):** 소리담은 **transcript를 보유** → SAIL의 일반론("articles are frequently missing")을 넘어 **본인 답변 문장에 오류 인라인 하이라이트 + 교정**(#1의 직접 산물). 그리고 진단에서 멈추는 SAIL과 달리 **루프를 닫는다**.

## 5. 기준 통합 매핑 테이블 (코드·기존 문서엔 없던 연결고리)

> 4개 기준(SAIL narrative ↔ FACT ↔ 세부진단서 42코드 ↔ WP코드 ↔ grammar_type)을 잇는 명시적 매핑이 코드/문서에 부재. 이 표가 Phase 1·3·4의 구현 기반.

| SAIL 리포트 섹션 | FACT 축 | 세부진단서(F1~F5) | WP 코드군 | grammar_type |
|------------------|---------|-------------------|-----------|--------------|
| Functions and Tasks | **F** Functions | F1 시제서술 · F4 돌발 | WP_S01~09 | — |
| Accuracy | **A** Accuracy | F5 Grammar(G1~5) | WP_A01~08 | TENSE·ARTICLE·SVA·PLURAL·PREP |
| Context and Content | **C** Context | F3 주제다양성 | WP_C01~07 | — |
| Text Type | **T** Text Type | F2 문단구성(11) | WP_T01~08 | — |
| (Delivery: 발음·유창성) | A 일부 | F5 Fluency·Pron(7) | WP_D01~04 | — |
| Tense (micro) | A | F1(11) + F5-G | WP_A | **TENSE** |
| Subject-Verb Agreement (micro) | A | F5-G Agreement | WP_A | **SVA** |
| Articles (micro) | A | F5-G Morphology | WP_A | **ARTICLE** |
| Plural (micro) | A | F5-G | WP_A | **PLURAL** |

**전진성 리포트 적용 예** (실측 약점 → 소리담 자산):
- 시제 혼용 → F1 / grammar_type TENSE → **시제 만능 아크**(past_* temporal_frame_control)
- 관사·복수·수일치 → F5-G / ARTICLE·PLURAL·SVA → 마이크로 문법 집중 드릴 (1주 Day 2~4, 이미 커리큘럼화)
- 단일 주제 문단 전개 → F2 / TextType → **skeleton paragraph(258룰)** + 쉐도잉 AL 모범답안 대조
- 주제 이탈 → F3 / Context → description_random `no_personal_anchor` 트리거

## 6. 단계 로드맵 (Phase)

| Phase | 항목 | 산출 | 비고 |
|-------|------|------|------|
| **0** | **통합 `ProficiencyReport` 스키마 정의** (§4.2) | 모든 후속의 전제 — 출처 무관 공통 입력 | 설계 결정 우선 |
| **1** | **리포트 엔진** — 모의고사 출력 SAIL화 (§4.5 #1·#2 + 인라인 교정) | grammar_type 태깅·정량화·본인답변 교정 → native 리포트 = **볼륨/데이터 엔진** | EF 평가단계 + DB 컬럼. MVP 핵심 |
| **2** | **외부 리포트 임포트** | 실제 SAIL/OPIc 리포트 붙여넣기 → 스키마 정규화 → **앵커 데이터** | 단일 왜곡 방어 시작 |
| **3** | **목표 등급 트레이너** (§4.1) | 갭분석 → **개인 훈련 생성**(기존 자산 호출) → 재측정 추적 = **사용자 북극성** | 1·2 위에 오케스트레이션 |
| **4** | **동적 매핑 캘리브레이션** (§4.3) | 정적 `WEAKNESS_MAP` → 모집단 기반 동적 처방 | 데이터 충분 후 |
| **+** | §4.5 #3~#8 (FACT micro·해석·병목·신뢰구간·다음등급예시·성장델타) | 리포트 풍부화 | 폴리시, 병행 |

**MVP**: Phase 0+1 = "통합 스키마 + 모의고사 SAIL식 리포트(마이크로 문법 + 본인 답변 인라인 교정)" → 데이터 엔진을 먼저 켜고, 그 위에 Phase 3 트레이너를 얹는다.
