# 평가 품질 테스트 — 소리담 데이터 이관 테스트

> 소리담의 완성된 모의고사 답변을 오픽톡닥에 이관하여 평가 파이프라인(4-Stage EF)의 품질을 검증한다.

## 1. 개요

### 목적
- 실제 사용자가 녹음한 **완성된 답변**으로 오픽톡닥의 평가 시스템을 실전 테스트
- 녹음을 새로 할 필요 없이, 소리담에 이미 있는 고품질 답변을 재활용
- 평가 결과(체크박스, FACT 점수, 등급, 코칭 피드백)의 품질 검증

### 테스트 대상 파이프라인
```
Stage A (mock-test-process)  → Whisper STT + Azure 발음 평가
Stage B (mock-test-eval)     → GPT-4.1 체크박스 평가 + 코칭
Stage C (mock-test-report)   → 규칙엔진 + FACT 점수 + 종합 리포트 + 성장 리포트
```

## 7. 트러블슈팅 & 재실행 가이드

### 첫 실행 시 겪은 문제와 해결

| # | 문제 | 원인 | 해결 |
|---|------|------|------|
| 1 | `mode 'practice'` check constraint 위반 | DB에 `training`/`test`만 허용 | `mode: 'test'`로 변경 |
| 2 | 소리담 음성 다운로드 400 | `mock-test-audio` 버킷이 **private** | Service key 인증 헤더 추가 (`apikey` + `Authorization`) |
| 3 | eval-judge 호출 401 (외부→EF) | EF가 JWT 검증 ON으로 배포됨. `Deno.env.get("SUPABASE_SERVICE_ROLE_KEY")`로 수동 인증 체크 하지만, **게이트웨이**가 먼저 JWT를 거부 | `--no-verify-jwt`로 재배포 (코드 내 수동 인증은 유지) |
| 4 | eval-judge 호출 401 (EF→EF 내부) | 동일 원인. admin-trigger-eval에서 호출해도 게이트웨이가 `Invalid JWT` 반환 | eval-judge, eval-coach, report 모두 `--no-verify-jwt`로 재배포 |
| 5 | admin-trigger-eval 컴퓨트 리소스 제한 | 14개 eval-judge를 순차 `await` → 타임아웃 | fire-and-forget 방식으로 변경 (응답 대기 X) |
| 6 | Q4만 `judge_completed`에 멈춤 | coach fire-and-forget 타이밍 이슈 | DB에서 `stt_completed`로 되돌린 후 재트리거 |

### 다음 실행 시 체크리스트 (원샷 성공 가이드)

#### 사전 조건
1. **EF 배포 상태 확인**: eval-judge, eval-coach, report가 `--no-verify-jwt`로 배포되어 있는지 확인
   ```bash
   # 필요 시 재배포
   SUPABASE_ACCESS_TOKEN=... npx supabase functions deploy mock-test-eval-judge --no-verify-jwt --project-ref rwdsyqnrrpwkureqfxwb
   SUPABASE_ACCESS_TOKEN=... npx supabase functions deploy mock-test-eval-coach --no-verify-jwt --project-ref rwdsyqnrrpwkureqfxwb
   SUPABASE_ACCESS_TOKEN=... npx supabase functions deploy mock-test-report --no-verify-jwt --project-ref rwdsyqnrrpwkureqfxwb
   ```
2. **admin-trigger-eval EF** 배포됨 (fire-and-forget 방식, `--no-verify-jwt`)
3. **소리담 접속**: Service key로 private 버킷 접근 필요

#### 실행 순서
```bash
# Step 1: 소리담 답변 데이터 다운로드
# tmp_soridam_answers.json 생성 (curl로 소리담 REST API 호출)

# Step 2: 시딩 스크립트 실행 (세션+음성+답변)
node scripts/seed-test-session.mjs

# Step 3: 평가 트리거 (admin EF 호출)
curl -s -X POST "https://rwdsyqnrrpwkureqfxwb.supabase.co/functions/v1/admin-trigger-eval" \
  -H "Content-Type: application/json" \
  -d '{"session_id":"<session_id>","admin_key":"opictalk-eval-2026"}'

# Step 4: 상태 모니터링 (1~3분 소요)
# psql로 eval_status, holistic_status, report 확인
```

#### 중요 파라미터
| 파라미터 | 값 | 비고 |
|---------|-----|------|
| mode | `test` or `training` | `practice` 사용 불가 |
| admin_key | `opictalk-eval-2026` | admin-trigger-eval 인증 |
| 소리담 Storage 접근 | Service key 필수 | private 버킷 |
| eval_status 시딩 값 | `stt_completed` | Q1은 `skipped` |

#### 스크립트 실행 후 eval 트리거 분리 이유
- `seed-test-session.mjs`는 **데이터 시딩만** (세션+음성+답변)
- 평가 트리거는 `admin-trigger-eval` EF로 분리 (EF→EF 내부 호출 필요)
- 외부에서 직접 eval-judge 호출 시 게이트웨이 JWT 검증 이슈
