## Part C: 오픽톡닥 구현 설계

### 스크립트 모듈의 본질 (3단계 목표)

```
[1차 목표] 나의 이야기 → 자연스러운 영어 스크립트
  - 문제: 응시자 대부분이 "내 이야기를 영어로 어떻게?" 에서 막힘
  - 해결: 한국어 키워드/간단 입력 → AI가 원어민이 실제 사용하는 자연스러운 영어로 생성
  - 핵심: 프롬프트(RCTF) + 등급별 규격서(script_specs) 품질

[2차 목표] 스크립트를 내 것으로 체화
  - 텍스트를 읽는 수준 → 입에 붙는 수준
  - 수단: 패키지(WAV+JSON) → 쉐도잉 5단계 훈련 (Step 1~4)

[3차 목표] 실제 원어민처럼 말하기
  - 스크립트 없이도 자연스럽게 나오는 상태
  - 수단: 쉐도잉 Step 5 (질문만 보고 자유 발화 + AI 평가)
```

**패키지 생성의 역할**: 1차(텍스트)와 2차(체화)를 연결하는 다리. 원어민 음성 + 문장별 타임스탬프가 있어야 체화 훈련이 가능.

**구현 우선순위**:
1. **스크립트 생성 품질** — RCTF 프롬프트 + script_specs가 핵심 ("진짜 원어민이 쓰는 말"이 나와야 함)
2. **입력의 간편함** — 한국어 키워드 몇 개만으로도 생성 가능
3. **패키지는 자연스럽게** — 사용자가 신경 쓸 필요 없이 자동 생성

### RCTF 프롬프트 아키텍처 (스크립트 생성 핵심)

> 소리담 3가지 방식(AI Generator / Template / Legacy)을 분석한 후, 제로 베이스에서 재설계한 아키텍처.
> 소리담의 RCTF 구조와 script_specs 60개 규격서는 활용하되, 프롬프트 조립 방식을 근본적으로 개선한다.

#### 핵심 문제 3가지

| 문제 | 설명 | 해결 전략 |
|------|------|-----------|
| **등급 드리프트** | GPT는 "좋은 영어"를 만들도록 훈련됨. IM2에 however를 끼워넣는 등 목표 등급을 초과하는 것이 가장 빈번한 실패 모드 | 등급 제약을 프롬프트 최상위에 배치 (primacy effect) + System에서 준수 원칙 강제 |
| **구어체 vs 문어체** | GPT 기본 출력이 written English으로 치우침. OPIc은 Speaking 시험 | 페르소나에 "spoken English" 강조 + 등급별 filler 규칙 명시 |
| **구조화된 출력** | 쉐도잉 5단계 훈련을 위해 paragraphs > slots > sentences > parts(T/E/F) 4계층 필요 | JSON Schema 강제 + T/E/F 분리 규칙 + 자가 검증 |

#### 설계 원칙 4가지

1. **제약이 생성보다 앞에 온다** — GPT는 프롬프트 앞부분에 더 주의를 기울임 (primacy effect). 등급 제약 → 구조 → 예시 → 입력 순서.
2. **하나의 예시가 천 마디 규칙보다 낫다** — 해당 등급의 완성된 JSON 예시 1개가 등급 캘리브레이션에 가장 효과적.
3. **script_specs가 진짜 IP** — 60개 규격서의 ACTFL 등급별 지식(허용 문법, 금지 표현, 담화 패턴)은 대체 불가능. 프롬프트는 이 규격서를 효과적으로 전달하는 매개체.
4. **System/User 역할 분리** — System=변하지 않는 규칙(캐싱), User=매 요청마다 바뀌는 동적 내용.

#### 프롬프트 구조 전체도


#### System Prompt 상세

```markdown
## ROLE

You are a certified ACTFL OPIc script coach (10+ years).
You transform learner input into natural spoken English scripts
that precisely match the target proficiency level.

Core principles:
1. LEVEL HONESTY — Never exceed the target level. A perfect IM2 script
   is better than an accidentally IH script.
2. SPOKEN ENGLISH — This is for a speaking test, not an essay.
   Use contractions, fillers, and conversational rhythm.
3. STORY PRESERVATION — Keep the learner's topics, details, and personality.
   Expand naturally, never replace.
4. SLOT FIDELITY — Each slot must fulfill its discourse function.

## CRITICAL RULE — LEVEL COMPLIANCE

You MUST strictly follow the LEVEL GATE in the user message.
The level constraints are absolute.
Never use grammar, vocabulary, or discourse markers from levels above the target.
A perfect script that stays within level boundaries is always better than
a script that accidentally uses higher-level expressions.

## T/E/F SEPARATION RULES

Every sentence is split into parts: template (T), expression (E), filler (F).

Definitions:
- template: Topic-independent sentence structure
  (works if you substitute any topic)
- expression: Topic-specific content
  (the learner needs to memorize this)
- filler: Discourse markers (Well, You know, I mean, Actually...)

Rules:
- T and E must alternate (T-E-T, E-T-E, T-E-T-E-T...)
- F is always a separate part, does not participate in T/E alternation
- Removing all F parts must leave valid T/E alternation
- Template must be ≥3 words of meaningful structure
- Expression must be a meaningful phrase (not a single common word)
- Punctuation attaches to the last part

Quality test: "Can I replace the topic in this template and it still works?"
✅ "I often go to ___" → park, gym, library all work
❌ "I go to the place with ATM machines near ___" → bank-specific

5 forbidden patterns:
1. Over-split template: ❌ [T "The "] [E "park"]
2. Weak expression: ❌ [E "visit"] (single common verb)
3. Topic info in template: ❌ [T "there on weekends"] (weekends = topic-specific)
4. Filler merged into template: ❌ [T "Well, I often go"]
5. Punctuation-only part: ❌ [T "."]

## CONTENT EXPANSION POLICY

Based on how much the learner provides:
- Level 1 (sufficient input): Restructure into slots, add natural details within their topic scope.
- Level 2 (partial input — keywords/fragments): Use their keywords as anchors, build slot content around them.
- Level 3 (no input): Generate a natural, relatable example using the question's topic.
In all cases: Follow level constraints and slot structure.

## KOREAN TRANSLATION

- Natural 의역, not 직역. Use 해요체.
- 3-layer consistency:
  sentences[].korean → slot.translation_ko → full_text.korean
  Concatenating all sentence.korean in a slot must equal slot.translation_ko.

## SELF-VALIDATION (before output)

1. Level check: Any forbidden grammar/vocabulary for this level?
2. T/E/F check: Any topic-specific words in template parts?
3. Filler check: Filler count within level-allowed range?
4. Korean check: sentence.korean concatenation = slot.translation_ko?
5. Text check: sentence parts concatenation = sentence.english?
```

#### User Prompt 조립 로직

**핵심: script_specs 규격서를 4개 섹션으로 분해하여 프롬프트의 최적 위치에 주입**

기존(소리담)은 규격서 전체를 `${script_spec}`으로 한 곳에 통째로 주입.
오픽톡닥은 규격서를 4개로 분해하여 각각의 역할에 맞는 위치에 배치:

```
script_specs 1개 (예: Description × IM3)
├── level_constraints  →  ① LEVEL GATE에 주입 (최상위 = 최강 주의력)
├── slot_structure     →  ② STRUCTURE에 주입 (구조 정의)
├── example_output     →  ③ EXAMPLE에 주입 (패턴 매칭용 JSON)
└── eval_criteria      →  System의 자가 검증에서 참조
```

```markdown

⛔ CRITICAL: Target level is {target_level}. NEVER use expressions from higher levels.

{level_constraints}
← script_specs.level_constraints 주입

{slot_structure}

## ④ INPUT

Question: {question_english}

Learner input: {user_story}

Generate a natural {target_level}-level spoken English script
```

#### Edge Function 조립 코드 (의사코드)

```typescript
// generate.ts — 프롬프트 조립 핵심 로직

async function assemblePrompt(
  question: MasterQuestion,
  targetLevel: string,
  userStory: string
): Promise<{ system: string; user: string }> {

  // 1. System Prompt (고정 — 캐싱됨)
  const systemPrompt = await loadTemplate('script_system');

  // 2. script_specs에서 4개 섹션 로드
  const spec = await supabase
    .from('script_specs')
    .select('level_constraints, slot_structure, example_output, eval_criteria, total_slots')
    .eq('guide_id', `${question.answer_type}_${targetLevel}`)
    .single();

  // 3. User Prompt 조립 (동적)
  const userPrompt = `
## ① LEVEL GATE — ${targetLevel}

⛔ CRITICAL: Target level is ${targetLevel}. NEVER use expressions from higher levels.

${spec.level_constraints}

---

## ② STRUCTURE — ${question.answer_type} × ${targetLevel}

${spec.slot_structure}

---

## ③ EXAMPLE

${JSON.stringify(spec.example_output, null, 2)}

---

## ④ INPUT

Question: ${question.question_english}
Question (Korean): ${question.question_korean}
Learner input: ${userStory || '(없음 — Level 3 확장 정책 적용)'}
Answer type: ${question.answer_type}
Total slots: ${spec.total_slots}

---

## ⑤ GENERATE

Generate a natural ${targetLevel}-level spoken English script
following all constraints above. Return JSON only.
`;

  return { system: systemPrompt, user: userPrompt };
}
```

```typescript
  response_format: {
    type: "json_schema",       // json_object보다 강력 — 스키마 100% 강제
    json_schema: scriptOutputSchema
  }
}
```

| 설정 | 값 | 근거 |
|------|-----|------|
| temperature | 0.8 | 1.0은 등급 드리프트 빈발. 0.8이면 자연스러우면서 제약 준수율 높음 |
| response_format | json_schema | json_object보다 강력. 출력 구조를 스키마로 100% 강제하여 형식 검증 불필요 |
| max_completion_tokens | 4000 | AL 8슬롯 200단어 + JSON 구조 + 한국어 번역 충분 |

#### 출력 JSON 구조 (4계층: paragraphs > slots > sentences > parts)

```json
{
  "paragraphs": [
    {
      "type": "introduction",
      "label": "서론",
      "slots": [
        {
          "slot_index": 1,
          "slot_function": "대상 소개",
          "text": "When I have free time, I usually go to a park near my house.",
          "translation_ko": "시간이 나면 보통 집 근처 공원에 가요.",
          "sentences": [
            {
              "index": 0,
              "english": "When I have free time, I usually go to a park near my house.",
              "korean": "시간이 나면 보통 집 근처 공원에 가요.",
              "parts": [
                { "type": "template", "text": "When I have free time, I usually go to " },
                { "type": "expression", "text": "a park near my house." }
              ]
            }
          ],
          "keywords": ["free time", "park", "near my house"]
        }
      ]
    },
    {
      "type": "body",
      "label": "본론",
      "slots": [...]
    },
    {
      "type": "conclusion",
      "label": "결론",
      "slots": [...]
    }
  ],
  "full_text": {
    "english": "When I have free time, I usually go to a park near my house. ...",
    "korean": "시간이 나면 보통 집 근처 공원에 가요. ..."
  },
  "word_count": 73
}
```

**4계층이 필요한 이유:**

| 계층 | 쉐도잉 훈련에서의 역할 |
|------|----------------------|
| paragraphs | 담화 구조 학습 (서론→본론→결론 흐름 인식) |
| slots | 슬롯 단위 의미 덩어리 (키워드 힌트 표시 단위) |
| sentences | Step 2~4 문장 단위 따라 읽기 (TTS 타임스탬프 매칭 단위) |
| parts (T/E/F) | Step 4 낭독 시 키워드 힌트, Step 5 실전 시 template만 표시 |

#### 소리담 대비 변경 요약

| 항목 | 소리담 | 오픽톡닥 |
|------|--------|---------|
| 프롬프트 방식 | 3가지 혼재 (AI Generator / Template / Legacy) | **1가지 통일** (System + User 분리형 RCTF) |
| spec 주입 | 전체를 한 곳에 통째로 `${script_spec}` | **4개로 분해** (level_constraints, slot_structure, example_output, eval_criteria) → 최적 위치에 주입 |
| 등급 드리프트 대응 | 자가 검증으로 사후 교정 | **LEVEL GATE(사전 방지)** + 자가 검증(사후 교정) 이중 장치 |
| System/User 분리 | 혼재 (V6 Template은 User만 사용) | **완전 분리** — System=고정 원칙(캐싱), User=동적 제약 |
| T/E/F 분리 | 방식 A만 3-type, 방식 B는 2-type | **3-type 통일** (T/E/F) — filler 독립 분리 필수 |
| 출력 구조 | 방식 A: paragraphs>slots>sentences>parts / 방식 B: flat slots | **paragraphs>slots>sentences>parts 4계층 통일** |
| JSON 출력 안정성 | json_object + 자가 검증 | **json_schema 강제** — 형식 검증 불필요, 내용 검증에 집중 |
| 예시 주입 | spec 안에 포함 (읽힐 수도 있고 안 읽힐 수도) | **독립 섹션**으로 분리 — JSON 형태로 패턴 매칭 강제 |
| temperature | 1.0 | **0.8** — 등급 드리프트 감소 + 충분한 자연스러움 |
| 프롬프트 언어 | 방식 A=한국어, 방식 B=영어 | **System=영어** (영어 생성 품질), 슬롯 기능명=한국어 |

### 테이블 설계 (7개)

#### scripts (통합 스크립트 — 스크립트 생성 + 튜터링 교정 공유)

```sql
CREATE TABLE scripts (
  id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id         UUID NOT NULL REFERENCES auth.users(id) ON DELETE CASCADE,
  question_id     TEXT NOT NULL REFERENCES master_questions(question_id),
  source          TEXT NOT NULL DEFAULT 'generate' CHECK (source IN ('generate','correct')),
  title           TEXT,
  english_text    TEXT NOT NULL,                -- full_text.english (검색/목록 미리보기)
  korean_translation TEXT,                      -- full_text.korean
  paragraphs      JSONB,                        -- 4계층: paragraphs > slots > sentences > parts(T/E/F)
  total_slots     SMALLINT,
  category        TEXT,                         -- topic_category 값
  topic           TEXT,
  question_korean TEXT,
  question_english TEXT,
  user_story      TEXT,                         -- 생성 시 한국어 스토리
  user_original_answer TEXT,                    -- 교정 시 학습자 원본 답변
  target_level    TEXT,
  answer_type     TEXT,
  ai_model        TEXT DEFAULT 'gpt-4.1',
  word_count      INTEGER,
  generation_time INTEGER,                      -- 초 단위
  key_expressions JSONB DEFAULT '[]',
  highlighted_script TEXT,
  -- 수정/재생성 관리
  status          TEXT NOT NULL DEFAULT 'draft' CHECK (status IN ('draft','confirmed')),
  refine_count    SMALLINT NOT NULL DEFAULT 0,  -- 수정 횟수 (최대 3)
  created_at      TIMESTAMPTZ DEFAULT NOW(),
  updated_at      TIMESTAMPTZ DEFAULT NOW(),
  UNIQUE(user_id, question_id)                  -- 사용자당 질문당 1개 (UPSERT)
);
```

- 인덱스: user_id, question_id, category, source, status
- UNIQUE(user_id, question_id): 재생성/재교정 시 기존 덮어쓰기
- `status`: draft(생성 직후) → confirmed(확정) — confirmScript SA에서 전환
- `refine_count`: 수정 횟수 추적 — 3회 초과 시 거부, 재생성 시 0으로 리셋
- **스크립트 모듈과 튜터링 모듈이 동일 테이블 공유**:
  - 스크립트에서 생성한 결과 → 튜터링에서 재활용 (재교정 불필요)
  - 튜터링에서 교정한 결과 → 스크립트에서 재활용 (재생성 불필요)
  - 사용자가 명시적으로 "다시 만들기" 할 때만 새로 생성
- source='generate': 스크립트 모듈에서 생성 (user_story 입력)
- source='correct': 튜터링에서 교정 (user_original_answer 입력)

#### script_specs (등급별 스크립트 규격서 — 60행, 4개 섹션 분리)

```sql
CREATE TABLE script_specs (
  id                SERIAL PRIMARY KEY,
  guide_id          TEXT UNIQUE NOT NULL,     -- 'description_IL', 'routine_IM2', 'advanced_14_AL' 등
  answer_type       TEXT NOT NULL,            -- 10가지 답변 유형
  target_level      TEXT NOT NULL,            -- IL, IM1, IM2, IM3, IH, AL
  total_slots       SMALLINT NOT NULL,        -- 등급별 슬롯 개수 (IL=3, IM1=4, IM2=5, IM3=6, IH=7, AL=8+)
  -- 4개 섹션 분리 (프롬프트 최적 위치에 개별 주입)
  level_constraints TEXT NOT NULL,            -- 언어 기준 + 금지 사항 → User ① LEVEL GATE에 주입
  slot_structure    TEXT NOT NULL,            -- 분량 기준 + 담화 패턴 + 슬롯 구조 → User ② STRUCTURE에 주입
  example_output    JSONB NOT NULL,           -- 완성된 JSON 출력 예시 → User ③ EXAMPLE에 주입
  eval_criteria     TEXT NOT NULL,            -- 평가 기준 → System 자가 검증에서 참조
  created_at        TIMESTAMPTZ DEFAULT NOW(),
  updated_at        TIMESTAMPTZ DEFAULT NOW()
);
```

- 10 answer_type × 6 level = **60행** 시드 데이터 (소리담 규격서 기반 재구성)
- **기존 `content` 1컬럼 → 4컬럼 분리**: 프롬프트 내 최적 위치에 개별 주입하기 위함
  - `level_constraints`: 허용 문법/어휘/필러 + ⛔ 금지 사항 (등급 드리프트 방지 핵심)
  - `slot_structure`: 분량 기준 + 담화 패턴 + [서론][본론][결론] 슬롯 구조
  - `example_output`: 해당 등급의 완성된 JSON 예시 (GPT 패턴 매칭용)
  - `eval_criteria`: 질문 대응 + 유형 고유 + 등급 준수 평가 기준
- Edge Function에서 guide_id로 조회 → 4개 컬럼을 User Prompt의 ①②③ 위치에 각각 주입
- 소리담 60개 규격서의 Markdown을 4개 섹션으로 파싱하여 마이그레이션

#### 위저드 5-Step (/scripts/create — 몰입형 immersive 레이아웃)

```
[/scripts/create]

  Step 1: 주제·질문 선택
    - 카테고리 선택: 일반(general) | 롤플레이(roleplay) | 어드밴스(advanced)
      · 카테고리 버튼: h-11 px-6 rounded-lg text-sm font-semibold (터치 타깃 충분)
      · 선택 시 primary-500 배경 + white 텍스트, 미선택 시 surface + foreground
    - 주제 그리드: 이모지 카드 (페이지네이션, 9개/페이지)
    - 질문 선택: 주제 선택 후 해당 주제의 질문 목록 표시
      · answer_type 뱃지 (색상 구분)
    - [다음] 버튼 (질문 선택 시 활성)

  Step 2: 경험 입력 + 목표 등급
    - 선택된 질문 표시 (영어/한국어)
    - 목표 등급 선택 (IL ~ AL, 프로필 등급 기본 선택)
    - 한국어 스토리/키워드 입력 (텍스트 영역, 선택 — 없으면 Level 3 확장)
    - [스크립트 생성] 버튼 (1크레딧 차감 안내)
    → SA `createScript` 호출 (DB 생성 + 크레딧 차감)
    → EF `generate` fire-and-forget 호출

  Step 3: AI 생성 중 (~30~40초)
    - 진행 표시: 스피너 + "나만의 스크립트를 만들고 있어요..."
    - **학습 콘텐츠 카드 로테이션** (5초 자동 전환):
      · opic_tips DB에서 target_level + answer_type 기반 필터링
      · 5개 카테고리: 만능 도입(opening), 필러(filler), 유형별 패턴(pattern), 감정 표현(emotion), 등급 팁(tip)
      · 각 카드: 카테고리 아이콘 + 타이틀 + 영어 표현 + 한국어 설명
      · 좌우 스와이프 또는 자동 전환
    - **폴링 메커니즘**: useQuery refetchInterval 3초 — english_text 채워지면 Step 4 전환

  Step 4: 결과 확인
    - **4계층 구조 렌더링** (paragraphs > slots > sentences > parts):
      · paragraph: 서론/본론/결론 섹션 헤더 (회색 구분선 + 라벨)
      · slot: 아코디언형 의미 덩어리 (slot_function 표시, 키워드 태그)
      · sentence: 영어 + 한국어 병렬 표시
      · parts: **T/E/F 색상 코딩**
        - template(T): foreground-secondary (회색 — 범용 구조)
        - expression(E): primary-600 테라코타 font-semibold (핵심 표현 — 암기 대상)
        - filler(F): foreground-muted italic (필러 — 자연스러움 보조)
    - **핵심 표현 요약**: key_expressions 목록 (하단 카드)
    - **단어 수/등급 정보**: word_count, target_level 뱃지

    3가지 액션:
    [확정하기]  → SA `confirmScript` (status: draft→confirmed) → Step 5
    [수정 요청] → 수정 (Refine) 플로우 (아래 참조)
    [재생성]    → 재생성 (Regenerate) 플로우 (아래 참조)

  Step 5: 패키지 생성 → 완료
    - Phase 1: EF `generate-package` (ElevenLabs TTS) → "원어민 음성 생성 중..." (~10~20초)
      · 음성 선택: Mark(남성) / Alexandra(여성) (기본: Mark)
      · 완료 시 ✅ "음성 생성 완료"
    - Phase 2: EF `generate-shadowing` (Whisper) → "쉐도잉 데이터 생성 중..." (~30~40초)
      · Whisper STT word-level → Levenshtein 매칭 → 타임스탬프 JSON
      · 완료 시 ✅ "패키지 준비 완료!"
    - 대기 중 학습 콘텐츠 계속 표시 (Step 3과 동일)
    - 완료 후:
      [쉐도잉 훈련하기] → /scripts/shadowing?packageId=xxx
      [내 스크립트]     → /scripts (내 스크립트 탭)
```

#### 수정 (Refine) 플로우

**용도**: "대체로 괜찮은데, 이 부분만 바꿔줘"

```
[Step 4 결과 화면에서 수정 요청 클릭]
  → 수정 지시 텍스트 입력 (예: "두 번째 슬롯을 좀 더 캐주얼하게 바꿔줘")
  → SA `refineScript` 호출 (refine_count+1 + 텍스트 필드 초기화)
  → EF `refine` fire-and-forget 호출 (기존 스크립트 + 수정 지시)
  → Step 3(생성 중, 폴링) → Step 4(결과 확인) 반복
```

| 항목 | 내용 |
|------|------|
| SA | `refineScript` — refine_count 증가 + english_text/paragraphs 등 초기화 (폴링 트리거) |
| EF | `refine` 핸들러 — 기존 스크립트 전체 + 수정 지시 → GPT → 수정된 JSON → DB 업데이트 |
| 크레딧 | **무료** (이미 생성 시 1크레딧 차감함) |
| 횟수 제한 | 확정 전 **3회** (refine_count ≤ 3, API 비용 관리) |
| 폴링 | queryClient.invalidateQueries → refetchInterval 3초로 결과 감지 |

#### 재생성 (Regenerate) 플로우

**용도**: "방향 자체가 마음에 안 들어" 또는 "스토리를 바꾸고 싶어"

```
[Step 4 결과 화면에서 재생성 클릭]
  → 스토리 수정 가능 (기존 user_story 프리필)
  → "현재 스크립트가 삭제되고 새로 생성됩니다" 확인 모달
  → [재생성 (1크레딧)] 클릭
  → SA `createScript` 호출 (UPSERT + 크레딧 차감)
  → EF `generate` fire-and-forget 호출
  → Step 3(생성 중) → Step 4(결과 확인) 반복
```

| 항목 | 내용 |
|------|------|
| SA/EF | 최초 생성과 동일 (createScript + generate) |
| 크레딧 | **1크레딧 차감** (새 생성과 동일) |
| 결과 | scripts UPSERT (덮어쓰기) + 기존 패키지 자동 삭제 |
| refine_count | **0으로 리셋** (새 스크립트이므로 수정 3회 다시 가능) |

#### 확정 후 (내 스크립트에서)

- 확정 후에는 수정 3회 무료 초기화 안 됨 → 다시 만들기 = 재생성으로 통일

### 스크립트 생성 플로우

```
[사용자 입력] question_id + target_level + user_story (선택)

[generate.ts]
  1. 구독 횟수 체크 (script_credits: 플랜 → 횟수권 순서)
  2. master_questions에서 질문 데이터 로드
  3. ai_prompt_templates에서 'script_system' → System Prompt 로드 (캐싱됨)
  4. script_specs에서 guide_id(answer_type_level) → 4개 섹션 로드
     └ level_constraints, slot_structure, example_output, eval_criteria
  5. User Prompt 동적 조립:
     ① LEVEL GATE ← level_constraints
     ② STRUCTURE  ← slot_structure
     ③ EXAMPLE    ← example_output (JSON)
     ④ INPUT      ← question + user_story + answer_type + total_slots
     ⑤ GENERATE
  6. GPT-4.1 호출 (system + user, json_schema, temp 0.8)
     → paragraphs > slots > sentences > parts(T/E/F) 4계층 JSON
  7. scripts UPSERT (user_id + question_id)
  8. 기존 패키지 삭제 (script-cleanup)
  → 스크립트 반환
```

- `guide_id`, `answer_type`, `target_level`, `total_slots`, `content` (TEXT)
- content 2,200~6,800자, 8개 Markdown 섹션 일관 구조

#### 마이그레이션 전략

**소리담 프롬프트 → 오픽톡닥 System/User 분리:**

```
소리담 (하나의 메시지에 모두 포함):
  [R] ROLE + 4대 원칙 + 내용 확장 정책
  [C] ${script_spec} 통째 주입
  [T] T-1~T-7 (슬롯 구성, 문장 분리, T/E/F, 번역, 키워드, 조합, 자가검증)
  [F] FORMAT + JSON 구조 + 예시
  [U] 질문 + 스토리/답변 + 등급 + 유형

오픽톡닥 (System/User 분리):
  System (고정, API 캐싱):
    ← [R] ROLE + 4대 원칙
    ← CRITICAL RULE (등급 준수 최우선)
    ← [T-3] T/E/F 분리 규칙 + 5 금지 패턴
    ← [R] 내용 확장 정책 3단계
    ← [T-4] 한국어 번역 원칙
    ← [T-7] 자가 검증 5단계

  User (동적, script_specs 4분할 주입):
    ← ① LEVEL GATE ← script_specs.level_constraints
    ← ② STRUCTURE  ← script_specs.slot_structure
    ← ③ EXAMPLE    ← script_specs.example_output
    ← ④ INPUT      ← 질문 + 사용자 입력
    ← ⑤ GENERATE
```

**generate와 correct가 동일 system_prompt 공유 가능한 이유:**
- System에 들어가는 내용(원칙, T/E/F 규칙, 확장 정책, 번역, 자가검증)은 두 모드에서 100% 동일
- correct만의 차이("학습자 보존", "교정 깊이")는 User Prompt의 ⑤ GENERATE 지시문에서 처리

#### `script_specs` 60행 마이그레이션

| 소리담 Markdown 섹션 | → 오픽톡닥 컬럼 | 비고 |
