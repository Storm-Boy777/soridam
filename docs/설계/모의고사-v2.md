## 0. 핵심 원칙

### 평가의 목적

```
evaluation_criteria_v2 (60행 기준표)의 역할:

"목표 등급 + 유형 기준으로, 이 응시자의 답변이 목표에 도달했는가?"

→ 도달:   "충족. 어떤 점이 잘 되었는가" (소견)
→ 미달:   "미충족. 어디가 부족한가" (소견) + "뭘 해야 하는가" (방향) + "핵심 병목" (약점)
```

### 3단계 흐름: 평가 → 종합 → 튜터링

| 단계 | 모듈 | 역할 | 비유 |
|------|------|------|------|
| **1. 개별평가** | `diagnose-v2` + `consult-v2` | 목표 등급 기준으로 현재 답변의 수준을 진단 | 의사 진찰 |
| **2. 종합평가** | `report-v2` | 전체 문항의 진단 결과를 종합하여 등급 판정 + 종합 소견 | 진단서 |
| **3. 튜터링** | 튜터링 모듈 | 진단 결과를 바탕으로 목표 달성을 위한 훈련 제공 | 처방 + 치료 |

### 모듈 간 역할 분리

- **diagnose-v2** = 체크박스 74개 pass/fail → **report-v2의 등급 판정**에만 사용
- **consult-v2** = transcript + criteria만 보고 **독립 판정** → 소견/방향/약점 생성
- **report-v2** = diagnose 체크박스 집계 + Rule Engine → 등급 판정 + 종합 소견
- **튜터링** = consult-v2의 약점(weak_points) + report-v2의 등급 → 훈련 과제 설계

```
diagnose-v2 (체크박스)  ──→  report-v2 (등급 판정)에만 사용
consult-v2 (소견)       ──→  transcript + criteria만으로 독립 판정. 체크박스 안 씀
```

### 불변 원칙

- ⛔ 기존 V1 코드 일절 수정하지 않음 (EF, 컴포넌트, SA, 타입, DB 테이블)
- ✅ V1 세션/음성/답변 데이터는 V2에서 읽기 가능 (평가 결과만 별도 테이블)

## 1. 파이프라인 구조

### V1 → V2 비교

| 항목 | V1 (현재 운영) | V2 (확정) |
|------|---------------|-----------|
| **Stage 수** | 4-Stage | **4-Stage (동일)** |
| **Stage A** | `mock-test-process` (STT+발음) | V1 공유 (동일) |
| **Stage B-1** | `eval-judge` (체크박스) | **`mock-test-diagnose-v2`** |
| **Stage B-2** | `eval-coach` (코칭) | **`mock-test-consult-v2`** |
| **Stage C** | `mock-test-report` (엔진+종합) | **`mock-test-report-v2`** |
| **GPT 모델** | B-1: gpt-4.1-mini, B-2: gpt-4.1 | **둘 다 gpt-4.1** |
| **프롬프트** | DB `evaluation_prompts` | **DB `evaluation_prompts_v2`** |

### 파이프라인 흐름도

```
V1:  process(STT+발음) → eval-judge(체크박스)     → eval-coach(코칭)          → report(엔진+종합)
V2:  process(V1 공유)  → diagnose-v2(체크박스)     → consult-v2(소견+방향+약점) → report-v2(엔진+종합)
                         ↓                         ↓
                         report-v2에만 전달          transcript+criteria만으로 독립
                         (등급 판정용)               (체크박스 안 씀)
```

### Stage별 상세

**Stage A: process (V1 공유)**
- Whisper STT + Azure 발음 평가
- V1 EF 그대로 사용 (수정 없음)

**Stage B-1: eval-v2 (체크박스 전용)**
- 체크박스 74개 pass/fail 판정만 수행
- 프롬프트: DB `evaluation_prompts_v2` (CO-STAR, key: diagnose_v2/diagnose_v2_user)
- 저장: `mock_test_evaluations_v2` 테이블 (체크박스 전용)
- 완료 시 → consult-v2 fire-and-forget 자동 체인

**Stage B-2: consult-v2 (소견 전용)**
- transcript + criteria(60행 기준표)만으로 독립 평가. 체크박스 안 씀
- 프롬프트: DB `evaluation_prompts_v2` (CO-STAR, key: consult_v2/consult_v2_user/consult_v2_schema)
- 저장: `mock_test_consults_v2` 테이블 (소견 전용)
- 전체 완료 시 → report-v2 fire-and-forget 자동 체인

**Stage C: report-v2 (등급 판정 + 종합)**
- diagnose-v2 체크박스 집계 → Rule Engine 7-Step → final_level
- GPT 2회 병렬: overview(종합 소견) + growth(성장 분석)

### evaluation_criteria_v2 (60행 — 확정, 검증 완료)

**구조**: `target_level` × `question_type` → `criteria_text`

```
6등급 (IL, IM1, IM2, IM3, IH, AL) × 10유형 = 60행
```

**criteria_text 섹션 구조** (CO-STAR 재설계 시 적용):
```
[1. 과제 기대]     ← 이 등급+유형에서 기대하는 수행
[2. 충족 기준]     ← 체크리스트 항목 (GPT가 pass/fail 판정)
[3. severity 판정] ← 같은 WP라도 등급별로 severity가 다름
[4. WP 체크 지침]  ← wp_type_priority에서 합쳐진 상태
  [4-0] 공통 안전장치 (5개, 전 유형 필수)
  [4-1] 1차 체크 (유형 고유, 공통과 중복 제거됨)
  [4-2] 2차 체크 (1차 문제없을 때)
  [4-3] 낮은 우선순위
  [4-G] 게이트키퍼
  [4-X] 과잉 체크 금지
  [4-I] 해석 포인트
```

## 3. consult-v2 프롬프트 설계 (확정, 검증 완료)

> 상세 설계 문서: `docs/설계/consult-v2-프롬프트-재설계.md`

### CO-STAR 구조

| 섹션 | 내용 |
|------|------|
| **C — Context** | OPIc 등급 체계 + 모듈 역할 + 재료 목록 |
| **O — Objective** | 5단계 목표 (transcript → 충족 기준 판정 → fulfillment → 소견 → 약점) |
| **S — Style** | 진단서 형식, 근거 인용 필수, 추상적 평가 금지 |
| **T — Tone** | 존대말 + 긍정→부족 순서 + 허용/금지 표현 목록 + ✅/❌ 예시 |
| **A — Audience** | 한국인 학습자, 내부 용어 차단 (FACT/WP 코드 등) |
| **R — Response** | 필드별 명세 (길이/개수/내용 기준) |
| **수행 절차** | Step 1~6 명시 + 발화 메타 보정 규칙 |
| **출력 예시** | Few-shot 1개 (description/IH/partial) |

### consult-v2 실행 흐름

```
1. consult_v2 system prompt 로드 (evaluation_prompts_v2, key: consult_v2)
2. consult_v2_user 템플릿 로드 (key: consult_v2_user)
3. consult_v2_schema 로드 (key: consult_v2_schema)
4. evaluation_criteria_v2에서 (target_level, question_type) 1행 로드
5. 변수 치환:
   - {criteria}              → 해당 등급×유형 기준표 (WP 체크 지침 포함)
   - {transcript}            → STT 텍스트
   - {target_level}          → 목표 등급
   - {question_type}         → 질문 유형
   - {question_text}         → 질문 원문
   - {duration}/{word_count}/{wpm}/{pronunciation_score}/{fluency_score}/{filler_ratio}
6. GPT-4.1 호출 (response_format: consult_v2_schema)
7. 출력: fulfillment + task_checklist[] + observation + directions[] + weak_points[]
```

### 출력 JSON 구조

```json
{
  "fulfillment": "partial",
  "fulfillment_summary": "1~2문장 요약",
  "task_checklist": [
    {"item": "[2. 충족 기준] 항목 그대로", "pass": true, "evidence": "transcript 인용"}
  ],
  "observation": "3~5문장. 강점 → 개선점 순서",
  "directions": ["2~3개. 실행 가능한 문장"],
  "weak_points": [
    {"code": "WP_S03", "severity": "moderate", "reason": "내부용 설명", "evidence": "transcript 인용"}
  ]
}
```

### 검증 결과 (2026-03-22)

- IH + description: partial 판정, 톤 안정, 금지 표현 0개 ✅
- IH + comparison (Q4 실데이터): unfulfilled 판정, WP_T03 severe 정확 ✅
- directions 2~3개 제어, weak_points 최대 3개 제어 ✅

### 탭 2: 세부진단표

- INT 4행 (20개) + ADV 5행 (42개) + AL 2행 (12개, IH/AL만)
- 체크 방향: ✅ = 충족, ☐ = 미충족 (학습자 관점)
- Performance 판정 5등급 한글화 (완전 충족/최소 충족/발전 중/출현 수준/간헐적)

### 탭 3: 문항별 평가

```
문항 카드 구조:
├─ Part A: 판정 — fulfillment + task_checklist[]    ← consult-v2 출력
├─ Part B: 소견 — observation (텍스트)               ← consult-v2 출력
├─ Part C: 방향 — directions[] + weak_points[]       ← consult-v2 출력
└─ Part D: 내 답변 — audio_url + transcript + meta   ← process(V1) + DB
```

### 탭 4: 성장리포트

5섹션: 등급 추이 차트 → 좋아진 점 → 부족한 점 → 유형별 변화 → 집중 훈련 포인트

- **F** (Functions & tasks): 수행할 수 있는 과제 유형

- **C** (Context & content): 대화 가능한 상황과 주제 범위

### Advanced Functions 5축
| # | 영문 | 한글 |
|---|------|------|
| 1 | Can narrate and describe in all time frames | 시제 통제 / 서술 정밀도 |
| 2 | Can consistently produce connected discourse of paragraph length | 담화 구조 |
| 3 | Can speak about a variety of topics, including current events | 주제 다양성 / 어휘 폭 |
| 4 | Can deal effectively with an unanticipated complication | 돌발 대처 |
| 5 | Can be readily understood by speakers not accustomed to non-native speakers | 이해도 / 유창성 / 발음 |
