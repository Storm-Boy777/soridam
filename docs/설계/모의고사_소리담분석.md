## 1. OPIc 시험의 본질

OPIc(Oral Proficiency Interview - computer)은 컴퓨터 기반 영어 말하기 시험이다.

**실제 시험 진행 방식:**
1. 컴퓨터가 **질문 오디오를 재생**한다 (AI 시험관이 질문을 "말한다")
2. 수험자는 질문을 **듣고** 이해한다
3. 녹음이 시작되면 수험자가 **음성으로 답변**한다
4. 답변이 끝나면 다음 문제로 넘어간다
5. 15문제를 모두 마치면 평가가 진행된다

**핵심 UX**: 질문 듣기 → (리플레이 기회) → 답변 녹음 → 다음 문제

소리담은 이 실제 시험 경험을 충실히 재현한다:
- **AVA**(AI Virtual Assessor)라는 가상 시험관 캐릭터
- 질문 오디오 재생 (TTS 사전 생성된 MP3)
- 재생 완료 후 자동 녹음 시작
- 5초 리플레이 윈도우 (1회 다시 듣기 기회)
- 문제별 시간 제한 (90~120초)

## 3. 전체 플로우 요약

### 사용자 관점 플로우

```
1. 모드 선택 (빈출/최신/커스텀)
2. 마이크 준비 확인
3. "시험 시작" 클릭
   ↓
4. Q1 자기소개 (120초, 평가 제외)
5. Q2~Q15 시험 문제 (각 90~120초)
   각 문제마다:
   a. AVA가 질문 오디오 재생 (시험관이 질문을 말함)
   b. 재생 끝나면 자동으로 녹음 시작
   c. 5초 안에 "다시 듣기" 가능 (1회)
   d. 다시 들으면 → 재생 후 다시 자동 녹음
   e. 답변 녹음 → "다음" 클릭 → 업로드
   f. 바로 다음 문제 오디오 재생
   ↓
6. Q15 답변 후 → 평가 대기 화면
   (문항별 진행률 실시간 표시)
7. 전체 평가 완료 → 결과 화면
```

## 4. Phase 1: 모드 선택 & 세션 생성

### 4.1 모드 선택 UI (`MockTestStartNew.tsx`)

**3가지 모드:**

| 모드 | 설명 | 문제 선택 로직 |
|------|------|---------------|
| `frequent` (빈출) | 시험 후기 빈도 기반 | 빈도 높은 순 정렬, 최근 10세션 중복 제거 |
| `recent` (최신) | 최근 출제 문제 | 최근 50개 후기 기반, 최근 5세션 중복 제거 |
| `custom` (커스텀) | 사용자 직접 선택 | CustomModeSelector UI에서 토픽/질문 선택 |

**활성 세션 감지:**
- 페이지 진입 시 기존 활성 세션(status = 'active')이 있는지 확인
- V6 세션 우선, V3 폴백
- 활성 세션 있으면 "이어하기" 버튼 표시

**시작 전 체크리스트:**
- 이어폰/헤드셋 권장
- 약 40분 소요
- 마이크 권한 필요

### 4.2 세션 생성 (`mock-test-create-v2` Edge Function)

**콤보 구성:**
```
세션 1개 = 5개 콤보 = 15문제

콤보 구성:
  Q1:  자기소개 (시스템, 120초, 평가 제외)
  Q2-Q4:  일반 콤보 1 (General, 각 90초)
  Q5-Q7:  일반 콤보 2 (General, 각 90초)
  Q8-Q10: 일반 콤보 3 (General, 각 90초)
  Q11-Q13: 롤플레이 콤보 (Roleplay, 각 90초)
  Q14-Q15: 어드밴스 콤보 (Advance, 각 120초)
```

**일반 콤보 3문제 패턴:**
```
문제 1: [Int] 묘사/설명 (describe)
문제 2: [Int/Adv] 상세/비교 (compare, routine)
문제 3: [Adv] 과거 경험/변화 (experience, change)
```

**문제 선택 알고리즘 (frequent 모드):**
1. `submission_combos`에서 combo_type별 빈도 집계
2. 빈도 높은 순 정렬
3. 최근 10개 세션에서 사용한 토픽 제외 (중복 방지)
4. 일반 콤보 3개는 서로 다른 토픽 강제
5. 부족하면 랜덤 보충

**기존 세션 정리:**
- 활성 세션이 있으면 `abandoned`로 만료
- 데드 세션 (Q≤1, 60분 이상 경과) 안전 삭제

**크레딧 차감:**
- `consume_mock_exam_credit` RPC 호출 (플랜 → 횟수권 순서)

## 5. Phase 2: 시험 진행 — 핵심 UX

> **이 섹션이 가장 중요하다.** OPIc 시험의 본질인 "질문 듣기 → 답변 녹음" 플로우를 상세히 기술한다.

### 5.1 시험 진행 상태 머신

```
[IDLE] ──(playQuestion)──→ [PLAYING]
                               │
                    audio.onended (첫 재생)
                               │
                               ▼
                   ┌──── [RECORDING] ◄────────────────┐
                   │     + canReplay = true            │
                   │     + 5초 카운트다운 시작           │
                   │                                   │
                   │  (사용자가 5초 내 "다시듣기" 클릭)    │
                   │         ↓                         │
                   │    녹음 중지                       │
                   │    [RE-PLAYING]                   │
                   │         │                         │
                   │    audio.onended (리플레이)         │
                   │         │                         │
                   │         └──→ [RECORDING]  ────────┘
                   │              canReplay = false
                   │
                   │  (사용자가 "다음" 클릭 or 시간 초과)
                   ▼
              [UPLOADING] ──→ 다음 문제 [IDLE]
```

### 5.2 질문 오디오 재생 (`playQuestion`)

```typescript
// MockTestSessionNewV6.tsx — 핵심 재생 로직 (간소화)

const playQuestion = (questionIndex: number) => {
  const question = session.questions[questionIndex];
  const audioUrl = question.audio_url;

  // 1. 기존 오디오 정리
  if (audioRef.current) {
    audioRef.current.pause();
    audioRef.current = null;
  }

  // 2. 상태 초기화
  setIsPlaying(true);
  setCanReplay(false);
  setPlayCount(0);

  // 3. 오디오 객체 생성 + 재생
  const audio = new Audio(audioUrl);
  audioRef.current = audio;

  audio.onended = () => {
    setIsPlaying(false);

    if (playCount === 0) {
      // ★ 첫 재생 완료 → 자동 녹음 시작 + 리플레이 윈도우
      startRecording();
      setCanReplay(true);
      startReplayCountdown(5); // 5초 카운트다운
      setPlayCount(1);
    } else {
      // ★ 리플레이 완료 → 자동 녹음 시작 (리플레이 불가)
      startRecording();
      setCanReplay(false);
    }
  };

  audio.onerror = () => {
    // 오디오 실패 시 WebSpeech 폴백
    playWithWebSpeech(question.question_text);
  };

  audio.play().catch(() => {
    playWithWebSpeech(question.question_text);
  });
};
```

**WebSpeech 폴백:**
```typescript
const playWithWebSpeech = (text: string) => {
  const utterance = new SpeechSynthesisUtterance(text);
  utterance.lang = 'en-US';
  utterance.rate = 0.9;
  utterance.onend = () => {
    startRecording();
    setCanReplay(true);
    startReplayCountdown(5);
  };
  window.speechSynthesis.speak(utterance);
};
```

### 5.3 리플레이 메커니즘 (5초 윈도우)

실제 OPIc 시험에서도 질문을 1번 더 들을 수 있다. 소리담은 이를 "5초 리플레이 윈도우"로 구현:

```
질문 재생 완료
    ↓
자동 녹음 시작 + "다시 듣기" 버튼 활성화
    ↓
5초 카운트다운 (5, 4, 3, 2, 1)
    ↓
[사용자가 5초 내 "다시 듣기" 클릭]     [5초 경과]
    ↓                                    ↓
  녹음 중지                          "다시 듣기" 비활성화
  질문 다시 재생                      녹음 계속 진행
    ↓
  재생 완료 → 자동 녹음 시작
  (리플레이 불가, 최대 2회 재생)
```

**핵심 포인트:**
- 첫 재생 후 **녹음과 동시에** 리플레이 카운트다운이 시작된다
- 사용자가 리플레이를 선택하면 **녹음을 멈추고** 질문을 다시 재생한다
- 리플레이 후에는 다시 듣기 불가 (최대 2회 재생이 한계)
- 리플레이를 선택하지 않으면 5초 후 자동으로 리플레이 비활성화

### 5.4 녹음 시스템 (RecordRTC)

```typescript
// 녹음 설정
const recorder = new RecordRTC(stream, {
  type: 'audio',
  recorderType: StereoAudioRecorder,  // WAV 포맷 (Azure 호환)
  mimeType: 'audio/wav',
  numberOfAudioChannels: 1,            // 모노
  desiredSampRate: 16000,              // 16kHz (음성 인식 최적)
  bufferSize: 8192,
});
```

**WAV 포맷을 사용하는 이유:**
- Azure Speech SDK 발음 평가가 WAV/PCM을 요구
- Whisper STT도 WAV를 잘 처리
- 압축 없이 원본 품질 유지 (발음 평가 정확도)

**마이크 시각화:**
```typescript
// Web Audio API로 실시간 볼륨 바 표시
const analyser = audioContext.createAnalyser();
analyser.fftSize = 64;
// → 20개 바로 마이크 입력 시각화
```

## 6. Phase 3: 답변 제출 & 업로드

### 6.1 클라이언트 → Edge Function (`mock-test-submit-v2`)

```typescript
// API 클라이언트 (mock-test-v6.ts)
async submitAnswer(
  sessionId: string,
  questionNumber: number,
  audioBlob: Blob,
  audioDuration: number
) {
  // 1. WAV blob → base64 변환
  const reader = new FileReader();
  const base64Audio = await new Promise<string>((resolve) => {
    reader.onload = () => resolve(reader.result as string);
    reader.readAsDataURL(audioBlob);
  });

  // 2. Edge Function 호출
  const { data } = await supabase.functions.invoke('mock-test-submit-v2', {
    body: {
      session_id: sessionId,
      question_number: questionNumber,
      audio_data: base64Audio,        // "data:audio/wav;base64,..."
      audio_duration: audioDuration,
      content_type: 'audio/wav',
    }
  });
}
```

**오디오 검증:**
```typescript
const MAX_AUDIO_SIZE = 50 * 1024 * 1024;  // 50MB
```

## 11. 오픽톡닥 현재 구현과의 차이점

### 11.1 핵심 차이 — ★ 반드시 개선 필요

### 11.5 오디오 관련 현황

**소리담의 질문 오디오 시스템:**
- `questions` 테이블의 `audio_url` 컬럼에 사전 생성된 TTS 오디오 URL 저장
- 시험 시작 시 15개 질문의 audio_url이 모두 세션 데이터에 포함
- 각 문제 진입 시 `new Audio(audioUrl).play()` → AVA가 질문을 "말함"
- 재생 완료(`onended`) → 자동 녹음 시작

**오픽톡닥의 현재 구현:**
- `questions` 테이블에 `audio_url` 컬럼 **471개 모두 URL 채워져 있음**
  - 패턴: `https://rwdsyqnrrpwkureqfxwb.supabase.co/storage/v1/object/public/audio-recordings/questions/{id}.mp3`
  - Storage 버킷 `audio-recordings/questions/`에 MP3 파일 존재
- `use-question-player.ts` 훅이 소리담과 동일한 패턴 구현 완료:
  - `play(audioUrl)` → 오디오 재생
  - `onPlaybackEnded` 콜백 → 자동 녹음 트리거
  - 5초 리플레이 윈도우 (`canReplay`, `replayCountdown`)
  - 1회 다시 듣기 (`replay()`, `hasReplayed`)
- `ava-avatar.tsx`에 `isSpeaking`/`isListening` props 구현됨
- **남은 작업**: 세션 컴포넌트(`mock-exam-session.tsx`)에서 이 훅들을 실제로 연결하는 것

---

> **결론**: 소리담 모의고사의 가장 핵심적인 UX는 "질문 오디오 재생 → 자동 녹음 시작 → 리플레이 윈도우"이다.
> 오픽톡닥은 이미 **DB 오디오 데이터(471개)**, **재생 훅(`use-question-player.ts`)**, **아바타 컴포넌트**가 모두 준비되어 있다.
> **남은 작업은 세션 컴포넌트에서 이들을 연결하는 것**이다.
