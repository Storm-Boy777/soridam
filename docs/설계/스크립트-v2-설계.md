# 스크립트 설계: Story Input & Fragment Mix

> 상태: Fragment Mix(V2) Phase A 구현 시작 대기 (2026-03-07~)
> 관련: `스크립트.md` (Story Input 현행 구현), `../의사결정.md`

---

## 1. 비전 & 네이밍

### 두 가지 스크립트 생성 방식

| | 영문 | 한글 | 설명 |
|---|---|---|---|
| **V1** | **Story Input** | **스토리 입력** | 질문마다 스토리를 직접 입력하여 생성 **(default)** |
| **V2** | **Fragment Mix** | **경험 조합** | 프로필 + 경험 조각을 조합하여 생성 **(베타)** |

### 운영 전략

```
Story Input (V1) — default, 현행 유지
   사용자가 질문을 선택하고 스토리를 직접 입력 → 스크립트 생성
   → 서비스 안정성 보장, 기존 플로우 그대로

Fragment Mix (V2) — 별도 개발, 베타 테스트용
   사용자 프로필 + 경험 조각 → 질문에 자동 매칭 → 스크립트 생성
   → V1과 병행 운영, 베타 사용자 평가 후 전환 여부 결정
```

### Fragment Mix의 목표
- **최소 노력으로 최대 등급** — 경험 조각 5~10개로 전체 기출 대응
- 사용자가 시험에서 **진짜 자기 이야기**로 말하게 만드는 것
- 사용자를 깊이 알수록 더 풍부한 스크립트가 나오는 구조

### 핵심 차이

```
Story Input:   질문 중심 — "이 질문에 뭐라고 답하지?"
Fragment Mix:  사용자 중심 — "이 사람은 누구인가? → 어떤 질문이든 이 사람의 이야기로"
```

## 2. 설계 진화 과정

논의를 거치며 설계가 점진적으로 발전한 기록:

| 단계 | 접근 | 한계 |
|------|------|------|
| AI 전문가 A | 스토리 재사용 + Question Clustering | 클러스터링이 기존 토픽과 중복 |
| AI 전문가 B | 만능 스토리 빌더 + 레고 블록 | 코어 텍스트 고정은 비현실적 |
| 초기 논의 | 토픽별 소재 생성 | 토픽마다 따로 만들어야 함 |
| 중반 논의 | 기출 역분석 Q&A | 설문조사처럼 느껴짐 |
| 후반 논의 | **사용자라는 사람을 먼저 파악** | - |
| **최종 결론** | **사용자 이해 + 경험 조각 조합** | 현재 설계 중 |

## 3. 핵심 구조: 경험 조각 조합 시스템

### 개념

사용자의 개별 경험들을 **조각(fragment)**으로 수집하고,
질문에 따라 1~2개 조각을 **조합**하여 스크립트를 생성한다.

```
사용자 이해 (온보딩)
    ↓
경험 조각 수집 (5~10개)
    ↓
AI가 경험 그래프 생성 (조합 가능성 매핑)
    ↓
질문 → 적합한 조각 1~2개 선택 → 조합 → 스크립트 생성
```

### 경험 조각 조합 예시

사용자 경험 조각:
```
A: 작년에 여수 여행
B: 영화 "왕과 사는 남자" 봄
C: 친구 민수랑 자주 만남
D: 배달 음식 좋아함
E: 신촌 원룸 자취
```

질문별 조합:
```
여행 질문      → A 단독: "작년에 여수 갔어요"
영화 질문      → B 단독: "왕과 사는 남자 봤어요"
여행+영화     → A+B: "여수에서 저녁에 영화 봤어요"
여행+친구     → A+C: "민수랑 여수 갔어요"
여행+음식     → A+D: "여수에서 게장이 대박이었어요"
친구+영화     → C+B: "민수랑 같이 봤어요"
집+음식       → E+D: "원룸에서 배달 시켜 먹어요"
집+친구       → E+C: "민수가 우리 집에 놀러 왔어요"
```

**경험 5개 → 조합으로 답변 10개 이상**

### 왜 이게 OPIc에 특히 강력한가

AL 답변의 핵심 = **extended discourse** (이야기를 길게 풀어내기)
- 경험 2개를 엮으면 자연스럽게 길어짐
- 억지로 늘리는 게 아니라 실제 에피소드가 두 개이므로 자연스러움
- "여수 갔을 때 민수랑 게장 먹고 영화 봤다" → 자연스러운 AL 담화

### 사용자에 대해 파악해야 할 정보 카테고리

| 카테고리 | 파악할 정보 | 기출 대응 |
|----------|------------|-----------|
| **나의 환경** | 어디 사는지, 동네, 집 구조 | description |
| **나의 일상** | 평일/주말 루틴, 출퇴근 | routine |
| **나의 취향** | 좋아하는 것, 취미, 여가 | description, routine |
| **나의 관계** | 가족, 친구, 동료 | description, past_special |
| **나의 소비** | 쇼핑, 음식, 기술 | routine, past_special |
| **나의 경험** | 기억에 남는 일, 문제/해결 | past_special, past_recent |
| **나의 과거** | 어릴 때, 이전과 달라진 것 | past_childhood, comparison |

### 온보딩 접근 방식

토픽별로 따로 묻는 것이 아니라, **"당신은 어떤 사람인가요?"**를 한번에 파악

```
온보딩 대화 (1회)
    ↓
사용자 프로필 + 경험 조각 수집
    ↓
이 프로필로 모든 토픽의 스크립트 생성 가능
```

예시:
```
A씨 프로필
- 28살 직장인, 서울 신촌 원룸 자취
- 퇴근하면 넷플릭스, 주말엔 카페에서 친구 만남
- 가끔 헬스장, 요리는 안 하고 배달
- 작년에 여수 여행, 최근에 에어컨 고장

이 프로필 하나로:
  집 → 원룸, 넷플릭스, 에어컨 고장
  자유시간 → 넷플릭스, 카페
  운동 → 헬스장
  음식 → 배달 음식
  여행 → 여수
  친구 → 카페에서 만남
```

## 6. 구현 구조

### 병행 운영 아키텍처

Story Input(V1)을 default로 유지하면서, Fragment Mix(V2)를 별도로 개발한다.
베타 테스트 후 사용자 평가 결과에 따라 전환 여부를 결정한다.

```
스크립트 모듈
├── [default] Story Input (V1) — 현행 유지, 수정 없음
│   ├── V1 위저드 (5단계)
│   ├── V1 Edge Function (scripts)
│   └── V1 프롬프트 (user_story 기반)
│
├── [베타] Fragment Mix (V2) — 별도 신규 개발
│   ├── 프로필 + 경험 조각 시스템 (Phase A)
│   ├── V2 전용 Edge Function + 프롬프트 (Phase B)
│   └── AI 챗봇 온보딩 (Phase C)
│
└── [공유] 두 방식이 공유하는 인프라
    ├── RCTF 슬롯 엔진 (script_specs 60행)
    ├── scripts 테이블 (generation_mode 컬럼으로 구분)
    ├── TTS 파이프라인 (scripts-package EF)
    ├── 쉐도잉 모듈
    └── questions DB (471개)
```

### Fragment Mix 구현 Phase

```
Phase A: 프로필 + 경험 조각 시스템 (기반)
   DB 테이블 + Server Actions + 입력/관리 UI
   → 사용자의 프로필과 경험 조각을 저장/관리하는 핵심 인프라
       ↓
Phase B: Fragment Mix 스크립트 생성 엔진
   V2 전용 Edge Function + 프롬프트 + 자동 매칭 + 생성 UI
   → 프로필/조각 기반으로 스크립트를 생성하는 핵심 기계
       ↓
Phase C: AI 챗봇 온보딩 (Phase A+B 동작 확인 후)
   대화 엔진 + 경험 자동 파싱 + 캐릭터 UI + 커버리지 시각화
   → Phase A에 데이터를 넣는 UX를 대화형으로 고도화
```

### Story Input(V1)과의 관계

| 영역 | 처리 |
|------|------|
| V1 위저드 / EF / 프롬프트 | **유지** — default로 계속 운영 |
| RCTF + 슬롯 엔진 (script_specs 60행) | **공유** — 두 방식 모두 동일하게 사용 |
| ai_prompt_templates | **추가** — V2 전용 프롬프트를 별도 행으로 추가 (V1 행 유지) |
| TTS / 쉐도잉 | **공유** — 변경 없음 |
| scripts 테이블 | **확장** — `generation_mode` + `fragment_ids` 컬럼 추가 |
| questions DB (471개) | **공유** — 매칭/매핑 용도로 활용 |

### 베타 테스트 전략

```
1. V1(Story Input)과 V2(Fragment Mix) 동시 제공
   → 사용자가 스크립트 생성 시 방식 선택 가능
   → 또는 특정 베타 사용자 그룹에만 V2 노출

2. 평가 기준
   → 생성된 스크립트 품질 (개인성, 구체성, 자연스러움)
   → 사용자 만족도 (어떤 방식이 더 좋았는지)
   → 학습 효율 (같은 시간 대비 커버하는 질문 수)

3. 전환 결정
   → V2가 우수: V2를 default로 전환, V1은 fallback 유지
   → V2가 미흡: V2 개선 후 재테스트 또는 V1 유지
```

## 7. DB 설계 (초안)

### 새 테이블: user_profiles (사용자 이해)

```sql
CREATE TABLE user_profiles (
  id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id uuid REFERENCES auth.users NOT NULL UNIQUE,
  raw_answers jsonb,              -- 온보딩 Q&A 원본 답변
  profile_structured jsonb,       -- AI 구조화된 프로필
  -- profile_structured 예:
  -- {
  --   "environment": { "home": "신촌 원룸", "area": "신촌역 근처" },
  --   "daily": { "weekday": "출퇴근, 넷플릭스", "weekend": "카페, 친구" },
  --   "preferences": { "hobby": "영화", "food": "배달", "exercise": "헬스" },
  --   "relationships": { "friends": ["민수"], "family": "부모님 경기도" },
  --   "past": { "childhood_home": "경기도 아파트" }
  -- }
  created_at timestamptz DEFAULT now(),
  updated_at timestamptz DEFAULT now()
);
```

### 새 테이블: user_experience_fragments (경험 조각)

```sql
CREATE TABLE user_experience_fragments (
  id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id uuid REFERENCES auth.users NOT NULL,
  fragment_text text NOT NULL,     -- "작년에 여수 여행 갔다"
  fragment_structured jsonb,       -- AI 구조화
  -- fragment_structured 예:
  -- {
  --   "event": "여수 여행",
  --   "when": "작년 여름",
  --   "who": ["민수"],
  --   "where": "여수",
  --   "emotion": "재미있었음",
  --   "details": ["게장 먹음", "밤에 해변 산책"],
  --   "topics": ["국내여행", "음식", "친구"],       -- 관련 토픽 자동 태깅
  --   "combinable_with": ["fragment_id_2", ...]     -- 조합 가능한 다른 조각
  -- }
  created_at timestamptz DEFAULT now()
);
```

### 기존 테이블 변경

```sql
-- scripts 테이블에 생성 방식 구분 + 경험 조각 참조 추가
ALTER TABLE scripts ADD COLUMN generation_mode text DEFAULT 'story_input';  -- 'story_input' | 'fragment_mix'
ALTER TABLE scripts ADD COLUMN fragment_ids uuid[];  -- Fragment Mix로 생성 시 사용된 경험 조각들
```

## 8. 프롬프트 설계

### Story Input (V1) — 유지
```
question + user_story + target_level → script
```

### Fragment Mix (V2) — 신규
```
question + user_profile + selected_fragments + target_level → script
```

### 프롬프트 context 예시
```
[User Profile]
- 28살 직장인, 서울 신촌 원룸 자취
- 취미: 넷플릭스, 가끔 헬스장
- 식생활: 배달 음식 위주
- 친구: 민수 (주말에 카페에서 자주 만남)

[Experience Fragments for this question]
- Fragment A: 작년 여름 여수 여행 (민수와 함께, 게장 먹음, 해변 산책)
- Fragment B: 에어컨 고장 (작년 여름, 수리 업체 불러서 해결)

[Question]
Tell me about a memorable trip you took recently.

[지시]
```

## 10. AI 캐릭터 인터뷰 시스템 (온보딩)

### 개념

캐릭터 기반 AI 챗봇이 사용자와 **친근한 대화**를 나누며 경험 조각을 수집한다.
설문이 아닌 **대화**. 사용자는 텍스트 또는 보이스로 답변한다.

### 핵심 설계

```
캐릭터 선택 (남/여)
    ↓
AI 인터뷰 시작 (채팅 UI, 텍스트 + 보이스 입력)
    ↓
빈도 기반 질문 우선순위 (시험후기 빈도 데이터 활용)
    ↓
사용자 답변 → LLM이 실시간 경험 조각 파싱 + 다음 질문 결정
    ↓
커버리지 실시간 시각화
    ↓
대화 종료 → 경험 조각 정리 → 프로필 완성
```

### 활용 기술 (이미 보유)

| 기술 | 오픽톡닥 현황 | 용도 |
|------|-------------|------|
| GPT-4.1 | Edge Function에서 사용 중 | 대화 엔진 + 경험 조각 파싱 |
| Whisper STT | 모의고사에서 사용 중 | 보이스 입력 |
| Gemini TTS | 스크립트 패키지에서 사용 중 | AI 캐릭터 음성 출력 (선택) |
| 기출 DB 460개 | questions 테이블 | 질문 우선순위 결정 |
| 빈도 데이터 | 시험후기 모듈 | 빈도 기반 우선순위 |

### 대화 흐름 예시

```
하루: "안녕! 나는 하루야 😊 오픽 준비 도와줄게.
      먼저 너에 대해 좀 알고 싶어! 요즘 어디 살아?"

유저: "신촌에 원룸 자취해요" (텍스트 or 🎤 보이스)

하루: "오 자취! 나도 자취하는데 ㅋㅋ 퇴근하면 주로 뭐 해?"

유저: 🎤 "그냥 넷플릭스 보면서 쉬어요"

하루: "넷플릭스 좋지~ 최근에 뭐 봤어?"

유저: "왕과 사는 남자요"

하루: "오 그거 재밌다던데! 어디서 봤어? 집에서?"

유저: "아뇨 여수 여행 갔을 때 봤어요"

하루: "여수! 누구랑 갔어? 뭐가 제일 기억에 남아?"

유저: 🎤 "친구 민수랑요. 게장이 대박이었어요"
```

→ 이 대화 한번으로 경험 조각 5개 자동 수집

### 빈도 기반 질문 우선순위

AI가 아무 질문이나 하는 게 아니라, **시험에 자주 나오는 영역부터** 커버한다.

```
[AI 내부 로직]

1. 빈도 데이터에서 상위 토픽 확인
   → 집(18%), 음식(15%), 영화(12%), 여행(10%), 운동(8%)...

2. 현재 수집된 경험 조각으로 커버되는 영역 확인
   → 집 ✅, 영화 ✅, 여행 ✅, 음식 ✅

3. 아직 커버 안 된 빈도 높은 영역 파악
   → 운동 ❌, 쇼핑 ❌

4. 자연스러운 다음 질문 생성
   → "평소에 운동 같은 거 해?"
```

### 커버리지 실시간 시각화

대화가 진행될수록 사용자에게 커버리지를 보여준다.

```
현재 경험 조각: 5개
기출 상위 30문제 대응: 22개 (73%)

[████████████████░░░░░░] 73%

💡 "여행 경험 하나만 더 추가하면 82%까지 올라가요!"
```

### 경험 재사용 (만능 경험)

수집된 경험 조각은 여러 토픽에 걸쳐 **재사용**된다.

```
경험: "민수랑 여수 여행 갔을 때 게장 먹음"

재사용:
  국내여행 질문 → 여수 여행 이야기
  음식 질문     → 게장 이야기
  친구 질문     → 민수랑 놀러 간 이야기
  영화 질문     → 여수에서 영화 본 이야기 (조합)
```

하나의 경험이 여러 토픽을 커버하므로, **경험이 쌓일수록 커버리지가 가속적으로 증가**

### 점진적 수집

온보딩에서 한번에 다 수집하지 않아도 됨:
- **1차 온보딩**: 기본 대화 5분 → 경험 조각 5~8개 → 기출 50~60% 커버
- **스크립트 생성 시**: 부족한 소재가 있으면 추가 질문 → 경험 조각 추가
- **사용할수록**: 프로필이 풍부해짐 → 커버리지 증가

## 11. 검증 로드맵

### 우려 사항 (자체 평가)

| 우려 | 내용 |
|------|------|
| 설계 비대화 | 스크립트 개선 → AI 챗봇 온보딩이라는 새 모듈로 확대됨 |

| 사용자 수용성 | 스크립트를 빨리 만들고 싶은 사용자가 온보딩 대화를 할지 미지수 |
| 개발 우선순위 | 튜터링 이관, 리브랜딩이 아직 남아있음 |

### 검증 완료

```
1단계: 프롬프트 검증 ✅ (2026-03-07)
   → V2 채택 확정. V1 대비 개인성/구체성/재사용성 모두 우수.
```

### 1단계 검증 결과 (2026-03-07 완료)

"집" 토픽(HSG_GEN_SEL_DESC_01), IH 등급, GPT-4.1로 비교:

**Story Input**: `"신촌에 원룸 자취. 퇴근하면 넷플릭스 봄"` (14글자)
**Fragment Mix**: 프로필(6항목) + 경험 조각 3개 (집 환경, 에어컨 고장, 친구 방문)

#### Story Input 생성 결과
```
I live in a small studio apartment in Sinchon. I chose this place because
it's really close to my office, so it's very convenient for me. My apartment
is in a lively area... The main room is where I sleep, eat, and relax.
There's also a small kitchen and a bathroom. Actually, it's just one room...
After work, I usually watch Netflix to relax...
```
→ 일반적인 원룸 묘사. 누구의 집인지 구분 불가.

#### Fragment Mix 생성 결과
```
Actually, I live alone in a small studio apartment in Sinchon, Seoul.
It's in a six-story building, and my place is on the fifth floor...
it's just a five-minute walk from Sinchon Station... The apartment has
a big window, so there's always a lot of sunlight... Inside, there's a bed,
a small desk next to it, and a TV that's mounted on the wall...
my friend Minsu sometimes visits, and since the room is small, we usually
sit on the bed together, order chicken, and watch Netflix.
```
→ "5층", "신촌역 도보 5분", "벽걸이 TV", "민수와 치킨" — 특정 인물의 집.

#### Fragment Mix 재사용성 검증 (routine 질문, 같은 프로필)
```
Question: "What do you usually do at home on a typical day?"

→ 넷플릭스("왕과 사는 남자"), 배달앱(치킨, 떡볶이), 민수 방문, 가끔 헬스장
→ description 스크립트와 일관된 캐릭터 유지 확인
```

#### 비교 결과

| 기준 | Story Input | Fragment Mix | 승자 |
|------|------------|-------------|------|
| 개인성 | 일반적 원룸 묘사 | "5층", "벽걸이TV", "민수+치킨" | **Fragment Mix** |
| 구체성 | "small kitchen" 등 일반 디테일 | "6층 건물 5층", "침대 옆 작은 책상" | **Fragment Mix** |
| 자연스러움 | 교과서적이나 자연스러움 | 필러 자연 배치 + 친구 에피소드 생동감 | **Fragment Mix** |
| 재사용성 | 질문마다 새 스토리 입력 필요 | 프로필 1회로 여러 질문 일관 커버 | **Fragment Mix** |

#### 핵심 발견
1. Fragment Mix가 압도적으로 구체적 — 경험 조각의 디테일이 그대로 반영됨
2. Fragment Mix의 진짜 강점은 재사용성 — 온보딩 1회로 모든 질문 커버
3. 토큰 비용: prompt +30% (838→1116) but 사용자 노력 1/N 감소
4. **결론: Fragment Mix 방식 병행 개발 확정** — Story Input은 default로 유지, 베타 테스트 후 전환 결정
