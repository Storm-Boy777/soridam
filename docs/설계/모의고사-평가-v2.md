## 1. 설계 철학

### V1 vs V2 근본 차이

| 관점 | V1 (현재) | V2 (재설계) |
|------|----------|------------|
| **설계 목적** | 채점 (Did you pass?) | 코칭 (How to improve?) |
| **사용자에게 먼저 보이는 것** | 통과율, 수치, 데이터 | 한줄 인사이트, 핵심 교정, 개선 답변 |
| **체크박스 역할** | 사용자 노출 (코드 직접 표시) | 내부 채점 전용 (사용자에게는 영역별 요약) |
| **교정 방식** | 오류 나열 (문법앱 수준) | Top 3 + wrong→correct→better 3단계 |
| **빠진 차원** | 구조 평가, 답변 개선, 등급 비교 | 전부 포함 |
| **톤** | ACTFL evaluator (학술적) | OPIc 코치 (실전적, 한국어) |
| **심층 분석** | 접힌 채 숨겨짐 | 코치 구조로 재편 + 기본 표시 |

### 핵심 원칙

1. **Insight-first**: 데이터보다 행동 가능한 인사이트가 먼저
2. **OPIc 등급 연결**: 모든 피드백이 "목표 등급으로 가려면"에 연결
3. **3개 제한**: 한 번에 3개 이상 고치면 학습 효과 하락 (인지 과부하)
4. **코치 톤**: evaluator가 아닌 coach, 학술 용어가 아닌 일상 한국어
5. **구조 > 문법**: OPIc 등급 상승의 핵심은 담화 구조 (sentence→paragraph→structured)

### 이중 목적 아키텍처

```
GPT 출력물
├── [내부 채점용] checkboxes → 규칙엔진 → FACT → 등급 판정
│   └── 사용자에게 노출하지 않음 (백엔드 전용)
│
└── [학습자 피드백용] insight + corrections + improvement + structure + ...
    └── 5-Layer 구조로 사용자에게 표시
```

> 기존 규칙엔진(V7)과 FACT 점수 체계는 **그대로 유지**. 변경하는 것은 "학습자가 보는 피드백"만.

## 2. 파이프라인 & Stage별 데이터 가용성

### 파이프라인 구조

```
[Stage A] mock-test-process — 변경 없음
    Whisper STT + Azure 발음 평가 + skip-detector
    ↓ fire-and-forget
[Stage B] mock-test-eval — 프롬프트 전면 교체
    GPT-4.1 개별 평가 (Q2~Q15, 각 1회씩 총 14회 호출)
    ↓ 전체 완료 시 fire-and-forget
[Stage C] mock-test-report — 프롬프트 전면 교체 + DB 저장 보강
    규칙엔진 7-Step (변경 없음) → 등급 확정 → GPT 종합 코칭 리포트 (V2)
```

### Stage별 데이터 가용성 (중요)

| 데이터 | Stage A | Stage B (개별) | Stage C (종합) |
|--------|---------|---------------|---------------|
| transcript (STT 결과) | 생성 | 사용 가능 | 사용 가능 |
| pronunciation (Azure) | 생성 | 사용 가능 | 평균으로 집계 |
| target_level (사용자 목표) | - | **사용 가능** | 사용 가능 |
| **확정 등급 (final_level)** | - | **없음** | **규칙엔진 후 확정** |
| FACT 점수 | - | 없음 | 규칙엔진 후 확정 |
| checkbox_type (INT/ADV/AL) | - | 사용 가능 | 집계됨 |
| 다른 문항 결과 | - | **없음** (자기 문항만) | **14문항 전체** |

> **핵심**: 개별 평가(Stage B)에서는 **등급이 확정되지 않는다**. `target_level`(사용자가 설정한 목표 등급, 기본값 IM2)만 사용 가능. 확정 등급은 Stage C에서 규칙엔진 실행 후에만 알 수 있다.

### Q1 자기소개 처리

- Q1은 시스템 고정 질문 (F-6)
- `submitAnswer()`에서 `eval_status: "skipped"` 처리
- Stage B/C 평가 대상에서 **완전 제외**
- 실제 평가 대상: Q2~Q15 (14문항)

### 스킵 판정 (skip-detector.ts)

| 단계 | 조건 | 결과 |
|------|------|------|
| Stage 1 | `audio_duration ≤ 15초` | STT + Azure + GPT 전부 스킵 |
| Stage 2 | `transcript.length ≤ 15자` | Azure + GPT 스킵 (STT는 실행) |
| Stage 3 | Whisper 환청 (이모지만 출력) | Azure + GPT 스킵 |

> 스킵된 문항은 `mock_test_evaluations`에 `skipped: true`로 기록되고, Stage C 규칙엔진에서 제외.

### 변경 범위

| 구성요소 | 변경 여부 | 상세 |

## 3. 체크박스 시스템 (변경 없음 — 참조용)

### 사전 정의 체크박스 ID

GPT가 임의 ID를 만드는 것이 아닌, `checkbox-definitions.ts`에 **사전 정의된 ID**로 평가.

| question_type | checkbox_type | 체크박스 수 | ID 예시 |
|--------------|--------------|------------|---------|
| description, routine | INT | 18개 | INT-1-1, INT-1-2, INT-2-1~3, INT-4-F1~5, INT-4-P1~4, INT-4-G1~2, INT-4-S1 |
| asking_questions | INT | 20개 | 위 18개 + INT-3-1, INT-3-2 |
| comparison, experience_* | ADV | 38개 | ADV-1-D1~3, ADV-1-N1~3, ADV-2-SP1~5, ADV-2-WO1~3, ADV-2-CD1~3, ADV-3-V1~3, ADV-5-F1~3, ADV-5-G1~5, ADV-5-PC1, ADV-5-P1~4 |
| suggest_alternatives | ADV | 42개 | 위 38개 + ADV-4-CR1~3, ADV-4-CD1 |
| comparison_change | AL | 6개 | AL-14-PS, AL-14-LS, AL-14-CS, AL-14-CD, AL-14-VB, AL-14-GA |
| social_issue | AL | 6개 | AL-15-AS, AL-15-MA, AL-15-SI, AL-15-CD, AL-15-VB, AL-15-GA |

### 체크박스 → FACT 매핑

| FACT | 매핑 체크박스 | 계산 |
|------|-------------|------|
| **F** (Functions) | INT-1(3) + INT-3(2) + ADV-1(7) + ADV-4(4) = 16개 | pass_count / total × 10 |
| **A** (Accuracy) | INT-4(12) + ADV-1(3) + ADV-2(8) + ADV-5(8) = 31개 | pass_count / total × 10 + 발음 보정(±1) |
| **C** (Context) | ADV-3(3) + ADV-1-L(2) = 5개 | pass_count / total × 10 - 유효문항 패널티 |
| **T** (Text Type) | INT-2(1~3점) + ADV-2-SP(2~9점) + CD 보정 | 단계별 스텝 점수 |

- 각 FACT: 0~10점, **total_score = (F+A+C+T) × 2.5 = 0~100점**
- 누적 로직: INT-2-3 PASS → INT-2-2, INT-2-1 자동 PASS / ADV-2-SP5 PASS → SP4~SP1 자동 PASS

### GPT 체크박스 출력 형식 (V1/V2 공통)

```jsonc
// GPT가 반환하는 형식 — 규칙엔진이 이 형식을 직접 파싱
{
  "checkboxes": {
    "INT-1-1": { "pass": true, "evidence": "적절한 어휘를 사용하여 주제를 전달함" },
    "INT-1-2": { "pass": true, "evidence": "질문의 의도를 정확히 파악함" },
    "INT-4-G1": { "pass": false, "evidence": "관사 누락 3회 (a/an/the)" },
    // ... question_type에 해당하는 모든 체크박스 ID
  }
}
```

> V2에서도 이 형식 그대로 유지. 규칙엔진 호환을 위해 체크박스 ID와 출력 구조는 변경하지 않는다.

## 4. 개별 평가 (Stage B) V2

### 4-1. GPT 출력 스키마

GPT에게 하나의 JSON으로 두 영역을 요청:
- **A. 내부 채점** (규칙엔진 입력용) — 기존과 동일
- **B. 학습자 피드백** (코칭용) — 완전히 새로운 구조

**중요: Stage B에서는 확정 등급이 없다. `target_level`(사용자 목표, 기본 IM2)만 참조 가능.**

```jsonc
{
  // ═══════════ A. 내부 채점 (백엔드 전용) ═══════════
  "checkboxes": {
    // checkbox-definitions.ts에 정의된 ID 그대로 사용
    // 형식: { "CB_ID": { "pass": boolean, "evidence": "한국어 근거" } }
    "INT-1-1": { "pass": true, "evidence": "..." },
    "INT-4-G1": { "pass": false, "evidence": "관사 누락 3회" }
    // ... question_type에 해당하는 전체 체크박스
  },
  "sentences": [
    // 기존과 동일: { index, text, group }
    // 교정 연결 + 인라인 표시용
    { "index": 0, "text": "I like go to internet cafe.", "group": 1 },
    { "index": 1, "text": "I play game and it is very fun.", "group": 1 }
  ],

  // ═══════════ B. 학습자 피드백 (코칭용) ═══════════
  "coaching": {

    // --- Layer 1: 즉시 인사이트 (3초) ---
    // ※ 확정 등급이 없으므로 "~등급에 머물고 있어요" 같은 표현 불가
    // ※ target_level을 기준으로 "목표(IM3)에 도달하려면" 형태로 작성
    "one_line_insight": "내용은 좋았지만 관사 누락이 반복되고 있어요. {target_level} 도달을 위해 문단 구조를 연습해보세요.",

    // --- Layer 2: 핵심 교정 3개 (10초) ---
    // OPIc 등급 영향도 기준으로 상위 3개만 선별
    "key_corrections": [
      {
        "sentence_index": 0,
        "original": "I like go to internet cafe",
        "corrected": "I like going to an internet cafe",
        "better": "I usually go to an internet cafe near my house after work",
        "why": "go to → going to: 동명사가 필요한 자리예요. 이런 기본 문법 오류는 등급 안정성을 떨어뜨립니다.",
        "impact": "high"  // high/medium/low — OPIc 등급 영향도 기준
      }
      // 최대 3개. impact 높은 순 선별.
    ],

    // --- Layer 3: 답변 개선 (30초) ---
    // corrected: 문법 오류만 수정
    // better: target_level 수준으로 재구성 (구조 + 연결어 + 구체적 예시)
    "answer_improvement": {
      "student_summary": "인터넷 카페에 자주 간다는 내용. 문장 3개 나열.",
      "corrected_version": "I like going to an internet cafe. I play games there. It's really fun.",
      "better_version": "I usually go to an internet cafe near my house after work. For example, last Friday I played my favorite game for about two hours. Because of that, I felt really refreshed and ready for the weekend.",
      "what_changed": [
        "문장 나열 → 시간순 이야기 구조로 변환",
        "구체적 예시 추가 (지난주 금요일)",
        "연결어 사용 (For example, Because of that)"
      ]
    },

    // --- Layer 4: 구조 + 영역별 분석 (2분) ---
    "structure_evaluation": {
      "has_intro": false,
      "has_body": true,
      "has_conclusion": false,
      "structure_score": 2,        // 1-5
      "structure_label": "문장 나열",
      // 1=단어나열, 2=문장나열, 3=문단시작, 4=구조적, 5=논리적
      "time_distribution": {
        "intro_pct": 0,
        "body_pct": 95,
        "conclusion_pct": 5
      },
      "story_density": {
        "example_count": 0,        // 구체적 예시/에피소드 수
        "detail_level": "low"      // low/medium/high
      },
      "tip": "도입부에서 '저는 보통 ~합니다'로 시작하고, 마무리에 '그래서 저는 ~가 좋습니다'를 넣어보세요."
    },

    "skill_summary": {
      "fluency":       { "score": 3, "label": "보통", "brief": "문장 사이 멈춤이 잦음" },
      "accuracy":      { "score": 2, "label": "부족", "brief": "관사, 동명사 오류 반복" },
      "content":       { "score": 3, "label": "보통", "brief": "주제 관련 답변이지만 구체성 부족" },
      "structure":     { "score": 2, "label": "부족", "brief": "도입/마무리 없이 문장 나열" },
      "pronunciation": { "score": 3, "label": "괜찮음", "brief": "대체로 이해 가능" }
    },
    // score: 1~5 (개별 평가는 5점 스케일)
    // label: 약함(1)/부족(2)/보통(3)/양호(4)/우수(5)

    // --- Layer 5: 심층 분석 (선택적 읽기) ---
    "deep_analysis": {
      "strengths": [
        "주제에 맞는 어휘를 사용했습니다 (internet cafe, game, relax)",
        "자연스러운 속도로 말했습니다"
      ],
      "weaknesses": [
        {
          "point": "관사(a/an/the) 누락이 3회 반복",
          "l1_cause": "한국어에 관사가 없어서 생기는 전형적인 L1 간섭",
          "level_impact": "등급 안정성을 떨어뜨리는 주요 원인"
        },
        {
          "point": "문장이 and로만 연결됨",
          "l1_cause": "한국어의 '~하고' 패턴을 영어에 그대로 적용",
          "level_impact": "상위 등급에서는 다양한 연결어(however, because, after that) 필요"
        }
      ],
      // ※ target_level 기준으로 피드백 (확정 등급이 아님)
      "target_gap": {
        "target": "{target_level}",  // 프롬프트에서 치환됨
        "key_gap": "문장 나열에서 문단 구조로의 전환이 필요합니다.",
        "example_at_target": "First, I usually go to the internet cafe near my office. For example, last Friday I went there after work and played my favorite game for about two hours. Because of that, I felt really refreshed."
      },
      "practice_suggestion": {
        "focus": "문장 → 문단 전환 연습",
        "method": "같은 주제로 3문장 답변 → 연결어를 넣어 1문단으로 합치기",
        "example_exercise": "'I like coffee. I drink it every morning. It makes me feel good.' → 'I really like coffee. For example, I drink it every morning before work. Because of that, I feel energized throughout the day.'"
      }
    }
  }
}
```

### 4-2. 프롬프트 설계 원칙

#### 페르소나

```
당신은 한국인 OPIc 학습자를 지도하는 전문 코치입니다.
학술 평가가 아닌, 학습자가 즉시 행동할 수 있는 피드백을 제공합니다.
```

#### 등급 관련 제약 (Stage B 한정)

```
- 이 단계에서는 학습자의 확정 등급을 알 수 없습니다.
- "현재 IM2입니다" 같은 등급 확정 표현을 절대 사용하지 마세요.
- 대신 학습자의 목표 등급({target_level})을 기준으로:
  "{target_level}에 도달하려면..." / "{target_level} 수준에서는..." 형태로 피드백하세요.
- better_version은 {target_level} 수준의 답변으로 작성하세요.
```

#### 프롬프트 핵심 지시

| 항목 | V1 지시 | V2 지시 |
|------|--------|--------|
| 체크박스 | "evaluate each checkbox as PASS or FAIL" | "체크박스는 내부 채점용으로만 평가. 학습자 피드백은 coaching 섹션에" |
| 체크박스 ID | checkbox-definitions.ts 정의 ID 사용 | **동일** (규칙엔진 호환 필수) |
| 교정 | "list all corrections" | "OPIc 등급에 가장 큰 영향을 주는 3개만 선별. 각각 wrong→correct→better 제공" |
| 심층 분석 | "linguistic_analysis, communicative_effectiveness..." | "strengths, weaknesses(+L1원인), target_gap(목표등급예시), practice(연습)" |
| 톤 | "professional evaluator tone" | "코치 톤. 한국어. '~해보세요', '~하면 좋겠어요'" |

#### question_type별 프롬프트 분기

V1과 동일하게 9개 프롬프트 유지 (8종 + 경험 3종 공유):

```
evaluation_prompts 테이블 key:
├── eval_description          → description (INT, 18CB)
├── eval_routine              → routine (INT, 18CB)
├── eval_asking_questions     → asking_questions (INT, 20CB)
├── eval_comparison           → comparison (ADV, 38CB)
├── eval_past_experience      → experience_specific, experience_habitual, experience_past (ADV, 38CB)
├── eval_suggest_alternatives → suggest_alternatives (ADV, 42CB)
├── eval_comparison_change    → comparison_change (AL, 6CB)
├── eval_social_issue         → social_issue (AL, 6CB)
└── eval_comprehensive        → 종합 리포트 (Stage C)
```

각 프롬프트의 구조:
```
[공통 프레임워크]
├── 내부 채점: 해당 question_type의 체크박스 ID 목록 명시 + 평가 지시
├── coaching 스키마: 위 JSON 구조 (모든 question_type 동일)
└── question_type별 특화 지시:
    ├── 묘사(description): 형용사 다양성, 공간 묘사 구조, "보통 ~가 있습니다" 패턴
    ├── 루틴(routine): 시간순 연결어, 습관 표현, 빈도부사
    ├── 비교(comparison): 비교 구문, 대조 구조, "On the other hand" 패턴
    ├── 경험(past_experience): 시제 일관성, 서사 구조, 구체적 에피소드
    ├── 비교변화(comparison_change): 시간 대비, 변화 표현, "used to / now" 패턴
    ├── 사회적이슈(social_issue): 논리 구조, 의견+근거, "I think ~ because" 패턴
    ├── 질문하기(asking_questions): 의문문 구조, 상황 설정
    └── 대안제시(suggest_alternatives): 문제+해결 구조, 제안 표현
```

### 발음 점수 해석

Azure accuracy_score → 학습자 친화 라벨. 프론트엔드 상수.

| 점수 범위 | 라벨 | 설명 |
|----------|------|------|
| 80~100 | 매우 좋음 | 원어민 수준으로 이해 가능 |
| 60~79 | 괜찮음 | 대체로 이해 가능 |
| 40~59 | 가끔 어려움 | 일부 단어에서 이해 어려움 |
| 0~39 | 자주 어려움 | 상대방이 이해하기 어려움 |

> IM 이상 등급에서 발음은 "이해 가능성(intelligibility)"만 중요 (ACTFL 기준). 점수보다 라벨이 적절.

## 8. 구현 순서

> 기존 테스트 데이터 삭제 전제. V1 하위 호환 불필요. 신규 기준으로만 구현.

### Step 1: 기존 데이터 정리 + DB 마이그레이션

```sql
-- 1. 기존 테스트 데이터 삭제
TRUNCATE mock_test_reports CASCADE;
TRUNCATE mock_test_evaluations CASCADE;
TRUNCATE mock_test_answers CASCADE;
TRUNCATE mock_test_sessions CASCADE;
TRUNCATE mock_test_evaluation_queue CASCADE;

-- 2. 컬럼 추가
ALTER TABLE mock_test_evaluations
ADD COLUMN IF NOT EXISTS coaching_feedback JSONB;

ALTER TABLE mock_test_reports
ADD COLUMN IF NOT EXISTS coaching_report JSONB,
ADD COLUMN IF NOT EXISTS recurring_mistakes JSONB;
```

### Step 2: V2 프롬프트 작성 (evaluation_prompts)

1. 개별 평가 V2 프롬프트 작성 (8종 + 경험 3종 공유)
   - 공통: checkboxes(사전 정의 ID) + coaching JSON 스키마
   - question_type별: 체크박스 ID 목록 + 특화 지시
   - 등급 제약: target_level 기준, 확정 등급 표현 금지
2. 종합 평가 V2 프롬프트 작성 (1종)
   - 확정 등급(final_level) 사용 가능
   - question_summaries V2 확장 형식 참조
3. evaluation_prompts 테이블 content UPDATE (9행)

### Step 3: EF 수정 + 배포

1. **mock-test-eval** (Stage B):
   - GPT 응답 파싱: checkboxes + coaching 분리
   - coaching_feedback 컬럼 저장
   - corrections, deep_analysis는 null (더 이상 사용 안 함)
   - prompt_version: "v2.0"

2. **mock-test-report** (Stage C):
   - `generateQuestionSummaries()` 확장: coaching_feedback 포함
   - coaching_report + recurring_mistakes 저장
   - overall_comments_ko: coach_diagnosis에서 추출

3. 양쪽 EF 배포

### Step 4: UI 재작성

1. 개별 평가 UI: 5-Layer 구조 (V2 전용, V1 폴백 없음)
2. 종합 평가 UI: 코치 진단 + 등급 비교 + 레이더 차트 + 반복 실수 + 실천 계획
3. FACT 점수: "말하기흐름(F) 8.5/10" + "합계 78.3/100"
4. 발음 → 이해도 라벨 변환

### Step 5: 검증

1. 테스트 세션으로 V2 평가 실행
2. GPT 출력 품질 확인:
   - 코치 톤 (evaluator 표현 없는지)
   - 개별 평가에서 확정 등급 언급 안 하는지
   - 체크박스 ID 정확성 (규칙엔진 호환)
   - better_version이 target_level 수준인지
   - 구조 평가 정확도
3. UI 렌더링 정상 확인
4. 규칙엔진 결과가 정상인지 확인
