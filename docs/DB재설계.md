> **목적**: 기존 master_questions 510행을 보강하여, 후기 제출 → 빈도 축적 → 확정 세트 승격 → 모의고사 활용으로 이어지는 자가 성장 DB를 설계한다.

> - [`이현석DB_분석.md`](이현석DB_분석.md) — 질문 원문 DB 분석 (1,011문항, 138세트)
> - [`강지완_기출분석.md`](강지완_기출분석.md) — 출제 빈도 통계 (680명)
> - [`기출매칭_분석.md`](기출매칭_분석.md) — 실제 기출 10건 매칭 검증

## 1. 핵심 컨셉

### 1.1 후기 제출 플로우

```
사용자가 "음악" 토픽 후기 제출 시:

① 확정 세트 먼저 보여주기
   ┌──────────────────────────────────────────────────┐
   │ ⭐ 확정 세트 A (127회 보고)                        │
   │   1. 좋아하는 장르/가수                             │
   │   2. 처음 좋아하게 된 계기                          │
   │   3. 라이브 음악 경험                               │
   ├──────────────────────────────────────────────────┤
   │ ⭐ 확정 세트 B (84회 보고)                          │
   │   1. 좋아하는 가수 묘사                             │
   │   2. 듣는 장소/시간                                │
   │   3. 취향 변화                                     │
   └──────────────────────────────────────────────────┘
   → 사용자: "이거다!" 클릭 → 끝 (원클릭)

② 없으면: "직접 선택"
   → 2번 질문 선택 (개별 질문 목록에서)
   → 3번 질문 선택
   → 4번 질문 선택
   → 이 조합이 새 콤보로 기록됨

③ 같은 콤보가 N회 이상 보고 → "확정 세트"로 승격 → ①에 추가
```

### 1.2 데이터 흐름

```
이현석 DB (시드)                    사용자 후기 (성장)
    │                                   │
    │  138 확정 세트                      │  새 콤보 축적
    │  1,011 개별 질문                    │  N회 이상 → 확정 세트 승격
    │                                   │
    └───────────┬───────────────────────┘
                │
        master_questions (개별 질문 풀)
                │
        confirmed_combos (확정 세트)
                │
        ┌───────┼───────┐
        │       │       │
     후기 제출  빈도분석  모의고사
     (선택UI)  (통계)   (출제풀)
```

### 1.3 핵심 원칙

| 원칙 | 설명 |
|------|------|
| **개별 질문 = 기본 단위** | DB의 최소 단위는 개별 질문. 세트는 질문의 조합으로 구성 |
| **확정 세트 = 빈도로 증명** | 같은 콤보가 N회 이상 보고되면 확정 세트로 승격 |
| **이현석 = 시드** | 이현석 DB의 세트는 이미 증명된 데이터 → 바로 확정 세트로 등록 |
| **빈도 = 자체 축적** | 강지완 외부 데이터 의존 X. 사용자 후기로 빈도 자체 축적 |
| **UX 개선 = 자동** | 확정 세트가 늘수록 "원클릭 선택" 비율 증가 → UX 자동 개선 |

### 3.1 테이블 구조

**현재**: `master_questions` 단일 테이블 510행

**논의**:

| 방안 | 구조 | 장점 | 단점 |
|------|------|------|------|
| A | 단일 테이블 유지 + 컬럼 추가 | 단순, 기존 코드 변경 최소 | 세트 관리가 별도 필요 |
| B | 질문 + 세트 2테이블 분리 | 세트 관리 자연스러움 | 기존 코드 수정 필요 |
| C | 주제 + 질문 + 세트 3테이블 | 가장 정규화 | 복잡도 증가 |

**결정**: ✅ **방안 C — 3테이블 (topics + master_questions + confirmed_combos)**

```
topics (주제 ~30행)
├── 집
│   ├── 질문1 (묘사)
│   ├── 질문2 (루틴)
│   └── ...
├── 음악
│   ├── 질문10 (좋아하는 장르)
│   └── ...
└── ...

confirmed_combos (확정 세트)
├── 세트1: [질문3, 질문7, 질문12]  (127회)
├── 세트2: [질문3, 질문8, 질문11]  (84회)
└── ...
```

- topics 테이블: 주제 메타데이터 (parent_topic, survey_type, 아이콘 등) 한 곳에서 관리
- master_questions: FK → topics, 개별 질문 풀
- confirmed_combos: 확정 세트 (이현석 시드 + 사용자 후기 승격)
- 3.2 parent_topic, 3.7 survey_type도 topics 테이블 컬럼으로 자연스럽게 해결

### 3.3 확정 세트(confirmed_combos) 구조

**개념**: 같은 질문 조합이 N회 이상 보고되면 "확정 세트"로 승격

**결정**: ✅

| 항목 | 결정 |
|------|------|
| **승격 기준** | **5회 이상 → 관리자 리스트업 → 관리자 검토/승인 → 승격**. 자동 승격 아님, 반자동(관리자 확인) |
| **세트 타입** | **별도 관리** — 일반(3문항), 롤플레이(3문항), 어드밴스(2문항)는 본질적으로 다른 구조 |
| **이현석 시드** | **초기 10회**로 삽입 → 바로 확정 세트로 표시, 사용자 데이터와 자연스럽게 섞임 |
| **저장 방식** | **`question_ids INTEGER[]` 정렬 배열** — 항상 오름차순 저장, `UNIQUE(combo_type, topic_id, question_ids)`로 중복 방지 |
| **정렬 순서** | **빈도순 → 동일 빈도 시 최근 보고순** — `ORDER BY report_count DESC, last_reported_at DESC` |

```
승격 플로우:
사용자 후기 제출 → 콤보 기록 → 동일 콤보 5회 이상 축적
  → 관리자 대시보드에 "승격 후보" 리스트업
  → 관리자 검토 → 승인 → confirmed_combos에 등록
  → 후기 제출 UI에서 "확정 세트"로 노출
```

### 3.5 sub_category / question_id 체계

**현재**: `sub_category` (Q2~Q15 고정), `question_id` (COM_APP_N_C1_01 코드)

**문제**: 일반 콤보 위치가 유동적이므로 Q2~Q10의 고정 번호는 부적절

**결정**: ✅

| 항목 | 결정 |
|------|------|
| **sub_category** | **Q11~Q15만 유지** (UI 표시용) — 일반(Q2~Q10)은 위치 유동적이라 번호 없음(NULL) |
| **question_id** | **삭제** — 기존 COM_APP_N_C1_01 코드 체계 폐기, 새 자동 증가 `id` (PK) 사용 |
| **세트 내 포지션** | **question_type이 포지션 역할** |
| **용도 구분** | `sub_category` = UI 표시 ("13번 문제"), `question_type` = 내부 로직 분기 |

### 3.6 영어 원문 (question_english)

**현재**: 없음 (한국어만)
**이현석 DB**: 영어 원문 보유

**결정**: ✅

| 항목 | 결정 |
|------|------|
| **컬럼 구조** | `question_english` (영어) + `question_korean` (자연스러운 한국어) + `title_short` (짧은 타이틀, UI 곳곳 활용) |
| **영어 원문 소스** | 이현석 DB 기반 + 중복 제거 |
| **저작권 전략** | 여러 기출 버전 교차 비교 → 동일 문장은 원문으로 간주 (그대로 사용) → 다른 부분만 최소 패러프레이징 |
| **패러프레이징 원칙** | 첫 문장(시작부) 동일 유지 (사용자 인지 핵심) + 후반부만 최소 패러프레이징 |
| **한국어 번역** | 이현석 것 복사 X, GPT로 자연스러운 새 번역 생성 |
| **짧은 타이틀** | 우리 자체 창작 (예: "좋아하는 음악/가수") — UI에서 원문 대신 간결히 표시할 때 사용 |
| **없는 질문** | NULL 허용 → 나중에 채움 |

```
저작권 전략:
① 여러 기출 버전 교차 비교 (이현석, 강지완, 기타 소스)
② 동일 문장 = 원문 확정 → 그대로 사용 (공공 영역)
③ 차이나는 부분 = 각자 복원한 것 → 최소 패러프레이징
④ 한국어/타이틀 = 우리 자체 생성
```

### 3.6.1 교차 비교 결과 (이현석 431 vs 강지완 565)

> **분석 도구**: `scripts/cross_match_v3.js` (boilerplate 제거 + 양방향 containment 유사도)
> **결과 파일**: `docs/OPIc 자료/cross_match_v3.json` (277KB)

**매칭 알고리즘**:
1. OPIc 정형화 문구 15+개 제거 ("You indicated in the survey that...", "I'd like to give you a situation..." 등)
2. 핵심 내용 단어만 추출 (최소 stop words)
3. 양방향 containment 유사도: (A→B 포함율 + B→A 포함율) / 2
4. 토픽 매핑 기반 — 같은 주제 내에서만 매칭

**분류 기준**:
| 등급 | 유사도 | 의미 |
|------|--------|------|
| A | ≥85% | 동일 텍스트 — ACTFL 원문으로 간주 가능 |
| B | 30~85% | 같은 기출 문제, 다른 표현 — 두 강사가 각자 기억해서 다르게 적은 것 |
| C | <30% | 이현석에만 존재 — 강지완 DB에 대응하는 질문 없음 |
